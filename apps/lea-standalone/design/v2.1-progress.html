Known gaps · deferred (not blocking)

Slice 6 · 6b — GitHub sharing (U3 + push UI) deferred pending a SettingsPage.tsx fix. The sharing half of the Filesystem slice — set a project's GitHub remote and Push to GitHub with a user-supplied token — is held because its token field belongs in Settings, and SettingsPage.tsx currently doesn't typecheck (it imports ApiKeyStatus/PermissionTier from lib/api which aren't exported, and sends a permission_tier the SettingsUpdate type doesn't define). The page will be reworked first, then the GitHub functionality added on top. 6a (browse/edit/export) is done + tested and unaffected. Fix direction: (1) repair SettingsPage.tsx + its lib/api.ts types so it compiles; (2) backend U3 — add a redacted github_token to settings.py (mirror the provider-key mask/validate/redact path; presence-only in settings_payload) + a read-only github_token() getter in config.py, plus PUT /git/remote (writes the existing projects.remote_url via store.update_project) and POST /git/push (a gitstore push helper that injects the token into the push URL only, never persisting it); (3) frontend — a Share bar in FilesystemTab.tsx (set-remote + explicit Push, target-confirm, disabled w/o token) + the redacted token field in the fixed Settings page (D43). Push is outward-facing → always an explicit button, never automatic.
SafeVerify can't audit a project proof that imports a sibling lemma. A project submission that does import Lea.<Project>.Foo fails SafeVerify with unknown module prefix 'Lea': safe_verify replays oleans inside its own Lake project (third_party/SafeVerify), whose search path has only its packages + Mathlib — not the main workspace's compiled Lea.* oleans (workspace/.lake/build/lib/lean). Loose proofs (single file, Mathlib-only imports) are unaffected; chaining proofs surfaced it (Q2/F3 testing). Fix direction: when the submission imports project modules, add the workspace build lib to safe_verify's LEAN_PATH (and ensure the sibling olean is built) in prover/lea/safeverify.py → verify_proof (the lake exe safe_verify invocation, cwd=SAFE_VERIFY_DIR).
MarkdownDoc: expand-to-modal for long Instructions / Memory (UX, low priority). When the user clicks Edit on an Instructions or Memory card, replace the Edit button with an expand icon that opens the editor in a full modal — so a long .lea/*.md doc is easier to read/edit than the small inline textarea. The modal mirrors the inline editor: same Cancel + Save buttons and PUT behavior. Fix direction: add an expand affordance + modal mode to MarkdownDoc.tsx (reuse the existing .modal-* styles from NewProjectDialog); the inline and modal editors share one draft/save() so either entry point works.
Blueprint graph: give the graph canvas more width (UX, low priority). The graph currently shares the Blueprint tab in a two-column layout (canvas + a 280px detail/legend rail), so a wide decomposition feels cramped. Give the canvas more room. Fix direction: in lea-v2.css, widen .bp-graph (raise the .pw-body max-width for the blueprint tab, and/or move the legend/detail rail below the canvas on wide graphs so the SVG spans full width); optionally make the detail panel a dismissible overlay rather than a fixed column.
Blueprint graph: distinguish "Proved" from "SafeVerify-audited" (correctness, low priority). A node's proved status today means the standalone Lean check passed (no sorry) on the stored code_step verdict — not a SafeVerify audit. SafeVerify is the stronger, audit-grade verdict (kernel replay + per-decl match + axiom whitelist) and should take precedence: a node is truly Proved only once SafeVerify passes; a Lean-check-clean-but-unaudited node should read as something like "check ✓ · audit pending." (Blocked in practice by the SafeVerify project-import gap above — sibling-importing proofs can't be audited yet.) Fix direction: add a SafeVerify verdict source feeding graph.py _base_status (a new status above proved, e.g. verified, or a verified: bool flag on the node); surface it in BlueprintGraph.tsx (STATUS_LABEL + a swatch) and the node detail. Resolve the LEAN_PATH gap first so project proofs are auditable.
