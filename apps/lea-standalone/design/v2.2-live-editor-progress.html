Design assumptions & open calls

Editing is modal in v1 (D62) — a session is either human-editing or has an active agent run, never both writing at once. This makes concurrent-write conflicts impossible rather than rare; a real human⇄agent merge model is Phase 3.
No Bubblewrap sandbox (D63) — local, single-user, and the agent already runs arbitrary Lean + bash, so a sandbox adds no coverage. ⚠️ Revisit trigger: if the adapter is ever exposed to untrusted remote users, re-introduce a container per session.
Biggest risk is the lean4monaco build, not the proxy (D68). It drags in Monaco + @codingame/monaco-vscode-* and needs Vite surgery (node polyfills, esbuild import-meta-url, worker static-copy, maybe COOP/COEP). Phase 1 (LF1) exists specifically to retire this early; the chunk is lazy-loaded so History-only users don't pay for it (mirrors #11's lazy Shiki).
One interactive lake serve per editing session is memory-heavy (D61). Fine for local single-user; mitigated by lazy start + idle reap. Pooling/sharing is a future optimization if pressure appears.
