# LeaUI v2.0.1 — Frontend State Refactor

**Status:** planned · **Goal:** turn the 578-line `App.tsx` god-component into a clean,
store-backed architecture **before** v2.1 (Projects), without changing any behavior.

This is a pure **refactor** — no new features, no UI changes. Every phase is
behavior-preserving and individually testable on the UI.

---

## 0. Why

`src/app/App.tsx` (578 lines) holds **five unrelated concerns** and prop-drills all of
it down to `ChatThread` / `Canvas` / `Sidebar`:

| Concern | State today (in App.tsx) |
|---|---|
| **Proof session** (the core blob) | messages, codeSteps, statusEvents, codeIndex, runStatus, runStatusById, currentRunId, isRunning, approvals, approvalBusy, error, safeVerify, editedPath + refs (eventSource, lastSeq, approvalCounter, runStatus) |
| **Sessions list** | sessions, selectedSessionId |
| **Model** | model, modelCatalog, modelFeatured, keyMissing |
| **Layout** | view, sidebarCollapsed, canvasCollapsed, canvasWidth, dragging + mainAreaRef |
| **Composer** | draft |

Three concrete problems:
1. **Five concerns, one file** — hard to read, hard to change.
2. **A 100-line SSE handler** (`attachStream`) of scattered `setState` calls.
3. **Prop-drilling** — `App` threads ~20 props into `ChatThread`; adding state touches App + every component in the chain.

v2.1 (Projects) will add state read in **three** places (sidebar Projects group, the
project window, the chat). On today's structure that makes App *worse*. Refactoring now —
while the state is fully understood and UI-tested — is the cheapest, lowest-risk moment.

## 1. Decision

- **State manager: [Zustand](https://github.com/pmndrs/zustand).** Tiny (~1 KB), one
  concept (`create`), no `<Provider>`, selector-based re-renders. Right fit for a
  single-user local app. **Not Redux** — its boilerplate/ceremony buys nothing at this
  scale; **not Context** — clunky for frequently-changing state.
- **Custom hooks** for cross-cutting UI concerns that don't need to be global (layout, the
  EventSource lifecycle).
- **Directory restructure** by layer: `stores/`, `hooks/`, `components/`, `lib/`, `ui/`.
- **Design the store shape fresh** from what the app actually needs — do NOT mechanically
  dump 24 `useState`s into a store 1:1.

## 2. Target architecture

```
src/app/
  App.tsx                 ← thin: compose stores/hooks + lay out the 3 panes (~60 lines)
  stores/
    proofSession.ts       ← Zustand: the proof-session blob + run/approval/verify actions
    sessions.ts           ← Zustand: session list + selection + load/reset
  hooks/
    useProofStream.ts     ← owns the EventSource lifecycle; maps SSE events → store actions
    useModel.ts           ← model + catalog + key check
    useLayout.ts          ← sidebar/canvas/resize/view + localStorage persistence
  components/             ← presentational only (ChatThread, Canvas, Sidebar, ModelPicker,
                            MarkdownMessage, SettingsPage, StatsPage)
  lib/                    ← pure logic (api.ts, timeline.mjs, codeDiff.ts, leanHighlight.mjs,
                            markdownParser.js, mathRenderer.js)
  ui/                     ← shadcn/Radix (unchanged)
```

Why this split:
- **proofSession** is the genuinely-shared blob — read by the header, `ChatThread`, and
  `Canvas`. Perfect store candidate. It also owns the SSE-derived state + the run/approval/
  verify/save actions.
- **sessions** is list/selection management; it coordinates with proofSession (loading a
  session fills the proof state).
- **useProofStream** keeps the imperative `EventSource` lifecycle (create/listen/cleanup)
  in a hook, calling store actions (`store.getState().onCodeStep(...)`). The store stays
  pure state+actions; the hook does the side-effecting subscription.
- **useModel / useLayout** are lighter, fewer consumers → plain hooks, not stores.

## 3. State ownership map

| Current `App.tsx` state | New home |
|---|---|
| messages, codeSteps, statusEvents, codeIndex | `proofSession` store |
| isRunning, currentRunId, runStatus, runStatusById | `proofSession` store |
| approvals, approvalBusy | `proofSession` store |
| error, safeVerify, editedPath | `proofSession` store |
| eventSourceRef, lastSeqRef, approvalCounterRef, runStatusRef | inside `useProofStream` / store |
| sessions, selectedSessionId | `sessions` store |
| model, modelCatalog, modelFeatured, keyMissing | `useModel` |
| view, sidebarCollapsed, canvasCollapsed, canvasWidth, dragging, mainAreaRef | `useLayout` |
| draft | stays local `useState` in `ChatThread` (not shared) |

## 4. `proofSession` store — shape sketch (not final)

```ts
const useProofSession = create((set, get) => ({
  // --- state ---
  sessionId: undefined, messages: [], codeSteps: [], statusEvents: [],
  codeIndex: 0, isRunning: false, currentRunId: undefined,
  runStatus: undefined, runStatusById: {}, approvals: [], approvalBusy: false,
  error: undefined, safeVerify: null, editedPath: undefined,

  // --- session lifecycle ---
  applyDetail: (detail) => set(/* fill from session_detail */),
  reset: () => set(/* clear for a new session */),

  // --- run lifecycle / SSE actions (called by useProofStream) ---
  startRun: (run) => set(...),
  onAssistantDelta: (text) => set(...),
  onMessage: (msg) => set(...),
  onCodeStep: (step) => set(...),
  onStatus: (ev) => set(...),
  onApprovalRequested: (a) => set(...),
  onApprovalResolved: (id, decision) => set(...),
  onDone: (status) => set(...),
  onError: (message) => set(...),

  // --- user actions ---
  decide: (decision) => ..., interrupt: () => ..., selectStep: (i) => ...,
  saveAndCheck: (content) => ..., verify: () => ...,
}))
```

The SSE handler stops being 100 lines of nested `setState`; each event is a named action.

## 5. Migration phases (each: behavior-preserving → build/tests green → you test on UI)

- **R1 · proofSession store.** Add Zustand. Create the store; move the proof-session blob +
  SSE actions into it. `App` calls store actions; `ChatThread`/`Canvas` subscribe to the
  store (drop those props). The biggest, highest-value slice.
- **R2 · useProofStream hook.** Pull the `EventSource` lifecycle out of `App` into a hook
  that maps events → store actions. `attachStream` leaves `App`.
- **R3 · sessions store.** Move session list/selection/load/reset; wire to `proofSession`.
- **R4 · useModel hook.** Move model + catalog + key-check.
- **R5 · useLayout hook.** Move sidebar/canvas/resize/view + localStorage.
- **R6 · lib/ + thin App.** Move pure modules to `lib/` (update imports); `App.tsx`
  collapses to composition + layout (~60 lines). Final pass.

## 6. Constraints

- **Behavior-preserving.** No feature/UX changes. If the UI behaves differently after a
  phase, that's a bug to fix, not an intended change.
- **No Projects yet.** v2.1 starts *after* this lands, on the clean store.
- **Test each phase on the UI** (interactive behavior is validated by hand), plus keep the
  pure-logic unit tests green (timeline, codeDiff, leanHighlight, markdownParser).
- **Design fresh.** Shape the store around the app's real needs, not the old `useState` list.

## 7. Risks & mitigation

- **Regressions in run/stream behavior** (the most complex part). → Migrate in small slices
  (R1→R2 split), keep actions 1:1 with the old `setState`s, re-verify each slice on the UI.
- **Import churn from the `lib/` move.** → Do it last (R6), mechanically, with a build check.
- **Over-globalizing state.** → Keep leaf/local state (e.g. composer `draft`) as `useState`;
  only genuinely-shared state goes in stores.
