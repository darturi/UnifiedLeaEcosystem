Design assumptions & open calls

Lean already solved the concurrency problem. Lean's server is a watchdog with one file-worker process per open file, explicitly designed for concurrent requests. Every real LSP client dispatches by JSON-RPC id and uri. Lea's daemon simply isn't one — it's a synchronous shim. So the rewrite is a correction, not a novel design, and it keeps one resident Mathlib.
Keep the 409 — redefine it as "at capacity". The Overleaf companion already fires runs concurrently and uses the 409 as its queue (leaApiClient.mjs:564-633). So: zero companion changes, and npm test -w apps/overleaf-extension staying green without modification is the proof we got it right. Removing the 409 without admission control would be worse than today — a 20-theorem doc would fire 20 simultaneous prover runs.
Concurrency inside one project directory is allowed. Serializing per project = the status quo with extra steps (all N theorems of one Overleaf doc live in one project). Worktrees and per-session subdirs both break Lean's srcDir := "proofs" module resolution. So: shared dir, different files. Updated by item 2: this used to read "per-repo lock + path-scoped git add" — measurement killed that. The shared dir stays (Lean requires it) but it stops being a repo: it becomes a re-materializable cache, and ordering moves to SQL, which has real transactions.
⚠️ Cross-phase ordering: 12 must land before 16. Item 2 stores one blob pointer per step and computes "before" as the previous step's after. That inference is sound only while a lock guarantees one writer per path — today's global cap-1 lock covers it, and item 12's path locks replace that. Item 16 flips the cap. If 16 lands before 12 there is a window with neither lock, and every diff in the system silently becomes wrong (before = someone else's write). This is the one ordering constraint that isn't self-evident from the phase numbers.
SQL is now the sole source of truth — so its concurrency correctness is load-bearing, not one item of 25. Item 1 is the foundation everything else stands on. Corollary from its measurement: WAL must never be enabled before a read-modify-write is inside db.write() — added alone it makes silent seq duplication worse (~110 → ~155 per 200), and it's exactly the kind of change that looks like a harmless perf tweak.
Subagent prompts compose the Lean core — they never replace it. opencode swaps the whole system prompt per agent; we can't. Every child must keep the hard rules (never modify a theorem statement, no sorry/axiom/native_decide). A replaced prompt could let a proof-candidate quietly prove a weakened statement — the exact cheat SafeVerify exists to catch.
Deferred on purpose: per-subagent token/cost accounting and budget UI. Depth and step caps stay (they're runaway guards, not accounting). Cost reporting is a reporting problem we can solve once the functionality exists — it isn't on the critical path and it isn't what makes this correct.
⚠️ Phase 5 is mandatory before Phase 6. useProofStream.ts:283-302 re-attaches on onerror with no backoff and no cap, and a browser EventSource can't read a 409 status. The moment the server can 409, that's an unthrottled request storm.

Phase 0 · Concurrency-safe substrate  ·  behavior-neutral

Harden the shared state while the global lock is still in place, so nothing can regress. No API surface, no user-visible change. If you want an even smaller first PR: 1 + 2 only.
1SQLite: db.write() — explicit BEGIN IMMEDIATE adapterTested ✓
store._next_seq does SELECT-max+1-then-INSERT, and its docstring asserted the safety argument: "Runs on the insert's own connection, so the read+write are atomic under SQLite's single-writer lock." That is false. Python's sqlite3 at its default isolation_level="" opens a transaction before INSERT/UPDATE/DELETE but not before SELECT — the SELECT runs in autocommit, holds no lock, and the read+write are not atomic. The failure is silent (not database is locked) and it corrupts the one thing seq exists for: the ORDER BY seq merge, and the latest-code_step lookup that session status derives from.
Measured, 8 threads × 25 inserts = 200, mirroring db.py exactly (deferred, rollback journal, and the busy_timeout=5000 that sqlite3.connect(timeout=5.0) already applies by default):
baseline (= main)0 errors 99–119 duplicate seqssilent
+ WAL only0 errors 151–158 duplicates⚠️ worse
explicit deferred BEGIN~145/200 lost 0 duplicatesloud, instant
BEGIN IMMEDIATE, busy_timeout=0 ~182/200 lost0 duplicatesloud
BEGIN IMMEDIATE + busy_timeout 0 errors200/200, contiguous 1..N✓
Two corrections to the original plan, both from measurement. (a) WAL is not part of the fix — the fix passes identically under the rollback journal, and WAL added alone to today's code pushes duplicates from ~110 to ~155, because it admits more concurrency and widens the race. It's an orthogonal throughput change and was dropped from this PR; db._open now carries a comment saying it must never land before the read-modify-writes move to write(). (b) It must be BEGIN IMMEDIATE, not BEGIN — a deferred BEGIN starts the SELECT as a reader and upgrades on the INSERT; SQLite does not run the busy handler for that upgrade, so it returns database is locked immediately regardless of busy_timeout (fails in 0.15s, not after a wait). busy_timeout is load-bearing for IMMEDIATE.
Shipped: new db.write() contextmanager (isolation_level=None + explicit BEGIN IMMEDIATE/commit/rollback) alongside the existing connect(). Scope was cut from the original plan: not a mechanical pass over ~40 sites — only the 2 real read-modify-writes (add_message, add_code_step) move, since flipping connect() globally would have silently dropped the implicit transaction from every multi-statement block. foreign_keys stays off (delete_project_cascade depends on it). _next_seq's false docstring is replaced with the actual precondition. Tests: new test_db_seq_concurrent.py (3 tests) — 10/10 runs fail on main, 5/5 pass with the fix, and the full 290-test suite is green.
2Demote git: SQL owns content, git becomes transport adapterPlanned · supersedes old 2 + 3 + 23
Absorbs the old "per-repo lock", "path-scope the diffs", and "shadow GIT_DIR" items — all three defended a store we're removing. Content moves to a content-addressed blob table (artifact_blobs(id, sha256, content) + code_steps.before/after_blob_id) — the direction docs/STORAGE-code-steps-vs-timeline.md already proposed, and the rule ghimport.py's D45 already follows for skills ("the row is the source of truth, not a live link"). Driver: a per-session git repo on one machine's filesystem is the single thing preventing a stateless, hostable app server.
Why not just fix the locking — measured. 24 concurrent writers, each writing its own distinct file into one repo (the shape project sessions already have, projects.py:142-149):
today18 × index.lock errors 24 writes → 1 commit6 commits contain other writers' files
path-scoped add alone18 errors 6/24 commits⚠️ 18 files never reach HEAD
per-repo lock alone0 errors 1/24 commits⚠️ 24/24 contaminated — silences the error, worsens the bug
lock + path-scoped0 errors24/24✓
The two halves are one change, not two — and either half alone is worse than today (scoping loses data; locking makes contamination total, because the working tree is written outside the lock). That kills any argument for landing a partial git fix early, which is what frees us to remove git as the store instead.
We never used git as git. No branch, no merge, no rebase — only content-addressing, show, and diff. sha256 + a blob table + difflib cover all three, with no subprocess per write and no repo-wide index mutex.
Schema — one pointer per step, not two. artifact_blobs(id, sha256 unique, content, created_at) + code_steps.after_blob_id. "Before" is the previous step's after_blob_id for that path, computed on read; the first step's before is null (the file didn't exist). This overrides docs/STORAGE-code-steps-vs-timeline.md, which proposes before_blob_id+after_blob_id — see the next two boxes for why that doesn't transfer. The write path shrinks: read → sha256 → insert → emit replaces four git subprocesses (add, diff --cached, commit, rev-parse). Blobs dedupe by sha256, so an unchanged file across steps costs one row, not N.
Capture rule: blob on FileChanged, read from disk. No new mechanism needed. events.py:56 already states the contract — FileChanged carries "path only; the bytes are read from disk / git show (filesystem-canonical, D3/D8), never shipped in the event" — and bridge.py:478 already reads the content back (via gs.snapshot) purely to stream it. So this is a substitution, not an addition, and it needs zero prover changes: D17 means the adapter never decodes which of the two write tools ran. Rejected alternative: a whole-dir hash scan at step end (opencode-style completeness) — unnecessary once the two points below hold.
⚠️ Why one pointer is sound — and the exact condition it depends on. opencode stores both endpoints (snapshot:{start,end}, packages/schema/src/session-message.ts:171-175) because start is an observed tree scan and they have no ownership model — an unlocked tree can drift under them, so both ends must be observed. We capture on an event under a lock instead: with only one writer per path, the previous blob provably is the disk content immediately before this write (handlers run sequentially, agent.py:729), so an observed "before" would be derived state twice over. The invariant: item 2's inference is sound only while a lock guarantees one writer per path — today's global cap-1 lock, or item 12's path locks. It must never be neither, so 12 lands before 16 flips the cap. (OpenHands sits at the opposite extreme: full old_content+new_content inline per edit, file_editor/definition.py:69-89 — ~2 whole copies of the file per edit, diff recomputed at render time, and their reconstruction path is latent but unimplemented. Neither extreme is ours.)
⚠️ Do NOT observe "before" at ToolCalled — it is a batching trap. It looks safe: ToolCalled is yielded at agent.py:600, the handler doesn't run until :729, and because run_events is a generator the adapter's read at that yield is guaranteed to precede the write (genuinely unlike opencode, whose hook fires but whose AI SDK runs the tool first). But a turn's ToolCalls are all emitted during the streaming loop before any handler runs — ToolCalled(A), ToolCalled(B), handler(A), handler(B). If A and B touch one path, B's "before" was read before A's write landed. That is opencode's empty-diff race in a different costume. This is the surviving form of old item 23's capture-point rule.
Close the bash hole via CheckResult — it's a live bug already. The prover's bash tool is explicitly told to do "file I/O outside the dedicated tools" (tools.py description, echoed at prompt.py:95), and FileChanged fires only for write_file/edit_file, only on .lean, only on success (agent.py:260-266). So a bash-created file emits no event. Today bridge.py:483 does if step_id: — a bash-created file that compiles fine has its verdict silently dropped: no code_step, invisible in canvas and timeline. Fix: when CheckResult (which carries a path, events.py:66) arrives with no step for that path, blob the file and create the step, attributed out-of-band. Bounded by design: a bash-written .lean nobody compiles is dead weight. Note this is a coverage hole, not a concurrency one — resource locks are a declaration, not a sandbox; nothing stops bash writing an undeclared path. Still far smaller than add -A, which contaminates by design rather than by misbehavior. Accepted: if bash edits a file that a later edit_file touches, the content is still correct (we read disk), only the attribution blurs — bash's change folds into that tool's diff.
Carve-outs — this is "demote", not "delete". (a) Lean needs real files on disk (lean_check compiles workspace/proofs/…; Lake can't read a blob table). The worktree stays — it's demoted from truth to cache, re-materializable from the DB. That's the actual invariant win: today a SHA in the DB points into a second store and the two can drift with no reconciliation. It also means hosting is "stateless read path + stateful Lean worker", not serverless — the worker still needs a disk and a multi-GB Mathlib. (b) Git stays as transport: ghimport's clone and push_to_github (D34) survive untouched; push becomes materialize-then-push. (c) Binaries don't belong in SQL — uploads.py commits PDFs to git today; keep the blob store behind an interface so they can move to object storage later without touching call sites.
Don't lose the old item 3's hazard in the migration — it's a live bug and survives the rewrite. bridge.py:238 _divergence_context diffs the whole repo, so a peer session's commit is handed to this run as "The human edited the proof files since your last turn" — showing someone else's theorem, which the agent may then go edit. Same on the canvas-save path (uncommitted_diff + routes/sessions.py:149); the modal editor lock is per-session, so it doesn't stop a human saving in session A while an agent drives session B of the same project. Blob-diffing makes the scoping natural (query this session's paths) but not automatic — it stays an explicit acceptance criterion, with test_bridge_divergence.py. Fails today.
Surface: 8 app modules (gitstore, bridge, projects, uploads, filesystem, ghimport, routes/{sessions,projects}) + 6 test files. Also update CLAUDE.md in the same PR — "Git owns proof content / proof bytes live in git, not the DB" is a documented invariant this deliberately inverts, and it must change where it becomes false, not later. Flag on the STORAGE doc's schema: its proposed patch, additions, deletions columns are derived from the before/after blobs — either compute on read or label them a cache explicitly, or it's _next_seq again (a comment asserting what nothing enforces).

Phase 1 · Make lsp_daemon.py a real LSP client

The heart of it. Lean's server is a watchdog with one file-worker per open file and is built for concurrent requests — so this buys real parallel checking, not just correctness, on one resident Mathlib. Still serialized in prod by the run lock, so the exercised path stays a strict subset of the new one — low landing risk.
5Dispatcher: per-request queues, id/uri routing, leases proverQueued
Rewrite the transport (~150 lines of a 317-line file). One reader thread dispatching responses by JSON-RPC id and notifications by uri into per-request queues; the shared self.queue is deleted, and with it the destructive purge and the message-dropping. Plus: _io_lock around _send (two threads on an unbuffered pipe can interleave Content-Length headers with bodies and permanently desync the framing); _state_lock for opened/version/calls; a per-URI lock for the same-document case (a Lean file worker holds one doc state per URI — versions can't fix that, a lock can); and leases so a stale daemon is retired from the registry and shut down only when its last in-flight check drains (today a mark_stale from any session calls shutdown() under another session's live check). Inject a _Transport seam so the dispatcher is testable without Lean.
6Bound the fallbacks: cold-check semaphore + lake-build lock proverQueued
lean_check swallows any LSP exception and falls back to a cold lake env lean (~88s, full Mathlib). The run lock caps that at one today — without a semaphore, one LSP hiccup under N runs spawns N Mathlib processes. And docs/PLAN-edit-save-latency.md:231 documents concurrent lake build against the shared workspace as an explicit non-goal → a global build lock keyed by lake_root, with mark_stale() called inside it. lean_check via the daemon is a different process, so it stays parallel with builds — that's the win.

Phase 4 · Path ownership + resource locks

The last shared thing nobody guards: files. Two runs in one project can pick the same filename; a subagent writing a candidate must never touch the canonical proof. This phase is what makes "many writers, one workspace" actually safe.
11Canonical path leases + temp candidate dirs adapterQueued
Each live run leases the canonical .lean paths it owns. Enforcement reuses the existing approval-gate seam: the adapter's gate(tool_name, args) denies a write_file whose path is leased by another live run, and the bridge immediately .send("deny") with no human in the loop — works for autonomous runs too, and needs no prover change. Children write to .lea/tmp/<run_id>/, which is git-ignored and never becomes a user-visible code_step unless the parent promotes it. Tests: test_project_path_ownership.py, test_candidate_workspace.py, test_subagent_transcripts_do_not_create_code_steps.py.
12Tools declare the resources they touch proverQueued
A tool declares which resources it needs (lean_file:<path>, lake_build:<root>, safeverify, lsp_uri:<uri>); the executor acquires them in sorted order (deadlock-free) from a FIFO lock manager with per-prefix timeouts — a lean_file wait is pathological at 30s, a lake_build wait is normal at 300s. The important part is the three-state default: a tool that declares nothing gets a tool-wide mutex, so an undeclared tool is safe by default and the failure mode of forgetting is slowness, not corruption. Tests: test_resource_locks.py.

Phase 7 · Subagents  ·  🟣 what this was all for

A run can spawn child runs. Everything here rests on seams that already exist in the prover — registry.build_toolset(selected) already scopes tools by name, skills.py already injects declarative markdown, events.py is already a typed stream. Each item still wants a short design pass before it's buildable.
18spawn_subagent tool + child activations proverDesign pass
A parent tool call starts a child activation with its own message list, its own candidate_dir, and a fresh context. Two independent recursion guards: a depth walk (default 1 — children can't spawn) and spawn_subagent denied by default in the child's toolset. Either alone would do; both means a bug in one isn't a fork bomb. Step and check caps are runaway guards, not accounting. Tests: test_subagent_depth_budget.py.
19Declarative agent profiles (markdown + frontmatter) proverDesign pass
One file per role in prover/lea/agents/: YAML frontmatter (model, tools, permission_mode, caps) + a markdown body that is the role's prompt. tools: maps straight onto the existing build_toolset(selected) — the list both filters and orders. Roles: coordinator · proof-candidate · mathlib-scout · error-triage · statement-check · refactor. Adding a role becomes writing a file, not writing code. Tests: test_subagent_profile_loading.py.
20Prompts compose the Lean core — never replace it proverDesign pass
Safety-critical, and where we deliberately diverge from opencode (which replaces the base prompt wholesale). Every child gets: shared Lean core (workspace · tools · tactic cascade · phrasebook · hard rules) → per-role head → volatile tail (workspace path, namespace, divergence context). A child that lost the hard rules could prove a weakened statement and report success. The layering is also cache-friendly: the shared core is a stable prefix across every child.
21Permissions compose by tightening only adapterDesign pass
A child's effective permissions = its own declared caps ∪ the parent's deny rules ∪ path rules. A child can never hold a permission its parent lacked — spawning is not a privilege-escalation path. Tests: test_subagent_permissions.py.
22Typed result envelope + transcript storage adapterDesign pass
A child returns a typed result (candidate path, check verdict, what it tried, why it stopped), not free prose the parent has to re-parse. The result id doubles as a resume handle and as the audit link from a promoted code_step back to the transcript that produced it — so "which attempt actually won, and what did it do" stays answerable. Transcripts are stored separately; they are not code_steps.
