# LeaUI v2.2 — Live Lean Editor in the Canvas (lean4monaco, owned): Architecture & Decisions

> Status: **living doc.** Source of truth for the **v2.2 Live Editor** — bringing a
> real Monaco + Lean-LSP editing experience (goals, hover, completion, diagnostics as
> you type, unicode abbreviations) into the Canvas, so a human can **edit and get
> language insights directly**, then hand the file to the agent. Continues the
> numbering of [`v2.1.1-factories-*`](./v2.1.1-factories-architecture.md) (which ends
> at **D58**) — Live-Editor decisions are **D59+**, so references never collide.
>
> **The one idea:** we do **not** run [`lean4web`](https://github.com/leanprover-community/lean4web)
> as a service. We take its **two reusable parts** — the `lean4monaco` client package
> and the *shape* of its LSP-over-WebSocket proxy — and **own them inside our own
> stack**: `lean4monaco` mounted in the Canvas, the proxy **re-implemented as a
> FastAPI WebSocket endpoint on :8001**. Everything else lean4web ships (its Node
> server, Yjs collaboration, the `Projects/` model, URL-hash sharing, Bubblewrap) is
> **dropped**. Owning it is what keeps the monorepo's one-backend invariant *and* buys
> the "features nobody's thought of yet" upside (agent-suggested inline diffs, shared
> InfoView, per-line SafeVerify markers — §5).
>
> **How to read this:** §0 is the canonical current design (read first). §1 is what
> lean4web actually is and what we take vs. drop. §2–§4 are the numbered decisions
> D59–D68 with rationale + rejected alternatives (the *why*). §5 is the extensibility
> payoff. §6 is the file layout; §7 the phased build plan. Live status in
> [`v2.2-live-editor-progress.html`](./v2.2-live-editor-progress.html).
>
> ### ⚠️ Scope of the current cut
> v2.2 ships **Phase 1 (read-only live insights)** and **Phase 2 (editable canvas)**.
> Phase 3 (features beyond lean4web) is **designed here but deferred** — kept as the
> record, planned per-item. The *active* decisions are **D59 · D60 · D61 · D62 · D63 ·

## 0. Current architecture at a glance

The Canvas today is a **read-only, Shiki-highlighted diff viewer of the agent's proof
snapshots** (`Canvas.tsx`), with a coarse `contentEditable` "✎ Edit → ▶ Run lean_check"
escape hatch and a manual "🛡 Run SafeVerify" button. Every check is a **batch
request/response** round-trip. There is **no live language intelligence in the
browser** — no goal state, hover, completion, or diagnostics as you type. That is the
gap v2.2 closes.

v2.2 adds a **second Canvas mode — "Live"** — that mounts a real Monaco editor
(`lean4monaco`) wired over a **WebSocket LSP** to a per-session `lake serve`. It extends
v2's invariants (in-process prover D-core, git-owns-proof-bytes D7/D8, one-backend-on-:8001,
session-status-is-derived D-status) to a live editing channel. (Decision refs in parens.)

- **We own the editor; lean4web runs nowhere (D59).** Take `lean4monaco` (client) +
  re-implement its proxy in FastAPI. No sidecar Node server, no third port.
- **The LSP proxy is a FastAPI WebSocket endpoint (D60).** `WS /api/sessions/{id}/lsp`
  pumps Content-Length-framed JSON-RPC both ways between the browser and a Lean server
  process, rewriting `file://` URIs between the browser's virtual URI and the session's
  real on-disk proof path. Reuses `lsp_daemon.py`'s spawn/framing helpers, but is a
  **pass-through**, not the daemon's batch `check()`.
- **The interactive server is per-session and separate from the agent's batch daemon
  (D61).** The existing `lsp_daemon` (one warm `lake env lean --server` per Lake root,
  used as a fast compiler behind the `lean_check` tool) is **left untouched**. The live
  editor gets its **own** `lake serve` per active editing session, lazily started on WS
  connect and idle-reaped.
- **Buffer ↔ disk/git is reconciled by an explicit save, and editing is modal vs.
  runs (D62).** While editing, source-of-truth is the Monaco buffer (streamed via
  `didChange`). On save it is persisted through the **existing** `POST /sessions/{id}/file`
  → git commit (`author=user`, D8/`gitstore.py`). v1 is **modal**: human-edit **xor**
  agent-run — never concurrent writes — so there is no split-brain.
- **Same Lake root as the agent (D64); no lean4web `Projects/` model.** `lake serve`
  runs in the session's own proof directory, so Mathlib + imports resolve **exactly**
  as the agent sees them.
- **No Bubblewrap (D63), no Yjs collaboration / URL sharing (D65).** Local, single-user,
  and the agent already executes arbitrary Lean + bash — sandboxing adds no coverage.
- **Monaco is a new Canvas mode, not a replacement (D66); the InfoView gets a panel
  (D67); the build is lazy-loaded (D68).** The Shiki snapshot stepper + SafeVerify badges
  (#10/#11) stay as the "History" mode; "Live" adds the editor. The `contentEditable`
  hack is retired by Live mode.

**Build status:** design only — nothing built. Phased plan in §7; live status in
`v2.2-live-editor-progress.html`.

## 1. What lean4web is, and what we take vs. drop

`lean4web` (the [`live.lean-lang.org`](https://live.lean-lang.org) playground) runs the
Lean server **server-side** and drives a browser Monaco editor over a WebSocket. Two npm
workspaces:

- **`client/`** — React + **Monaco** via the [`lean4monaco`](https://github.com/hhu-adam/lean4monaco)
  package. That package reuses the real Lean4 VSCode extension code compiled for the
  browser: live diagnostics, hover, **goal state / InfoView**, completion, unicode
  abbreviations (`\to → →`), go-to-definition. It opens an LSP connection to
  `wss://<host>/websocket/<project>` (`client/src/App.tsx` — `options.websocket.url`).
- **`server/index.mjs`** — a ~400-line Express + `ws` app that is essentially an **LSP
  proxy**: per WebSocket connection it spawns one `lake serve` (sandboxed with
  **Bubblewrap** in prod), pipes JSON-RPC between the socket and the process stdio
  (`vscode-ws-jsonrpc`), and rewrites `file://` URIs between client and server paths
  (`urisToFilenames` / `FilenamesToUri`). Also: static "Projects" (each a Lake project
  with `leanweb-config.json`, keyed by folder name in the URL) and optional Yjs/WebRTC
  collaboration.

**Take:**
1. **`lean4monaco`** (the client editor package) — the crown jewel. Mount it in the Canvas.
2. **The proxy *shape*** (spawn `lake serve` → pump framed JSON-RPC → rewrite URIs) —
   **re-implemented** in FastAPI, not adopted as code.

**Drop:** the Node/Express server, Yjs/WebRTC collaboration, URL-hash sharing, the
`Projects/`+`leanweb-config.json` model, and Bubblewrap. Each either duplicates something
we already own (backend, per-session git store, warm `lsp_daemon`, SafeVerify) or is
scoped out for a local single-user tool.

## 2. Ownership & the proxy

### D59 — Own the editor; do not run lean4web as a sidecar service
Integrate `lean4monaco` into our client and re-implement lean4web's proxy as a FastAPI
WebSocket endpoint. lean4web's own server never runs.
- **Why:** the monorepo invariant is **two front ends, one backend on :8001** (CLAUDE.md).
  A sidecar Node server would be a third service that duplicates process management,
  reconciles awkwardly with our git-backed session file model, and cedes control of the
  seam where the interesting future features live (§5). The proxy's *real* logic is ~80
  lines; owning it is cheap and strategic.
- **Rejected:** *run `lean4web/server` as-is, mount our proof dir as a "Project."* Fastest
  to a working demo, but adds a second backend language/process, forks the file-ownership
  model, and violates the one-backend rule. Kept only as the Phase-0 spike fallback if the
  Python proxy hits an unforeseen protocol wall.
- **Consequence:** all backend work is in the adapter; the client gains one heavy dep
  (`lean4monaco`) and Vite config (D68).

### D60 — The LSP proxy is a FastAPI WebSocket endpoint on :8001
Add `WS /api/sessions/{id}/lsp`. It bidirectionally pumps **Content-Length-framed
JSON-RPC** between the browser socket and a Lean server process's stdio, and rewrites
`file://` URIs between the browser's virtual document URI and the session's **real
on-disk proof path** (mirrors lean4web's `urisToFilenames`/`FilenamesToUri`).
- **Why:** FastAPI/Starlette speaks WebSockets natively; `lsp_daemon.py` **already** has
  the process-spawn + LSP framing (`Content-Length` read/write, `initialize` handshake)
  we need, so the proxy is a **pass-through sibling** of the daemon — not a re-derivation.
  Keeping it in the adapter means the browser talks only to :8001.
- **Rejected:** *reuse `lsp_daemon.check()` and stream its diagnostics.* That method is a
  **batch compiler** (send `didChange`, wait for `publishDiagnostics` + empty
  `fileProgress`, return a formatted string). A live editor needs the *raw* bidirectional
  LSP stream (hover, completion, goals, incremental diagnostics), not a cooked string.
  Share the framing helpers; don't route through `check()`.
- **Consequence:** a new `adapter/app/lsp_proxy.py` (pump + URI rewrite) + a route; the
  browser opens the socket with `lean4monaco`'s `websocket.url` pointed at it.

### D61 — One interactive Lean server per active editing session, separate from the agent's batch daemon
The live editor gets its **own** `lake serve` (or `lake env lean --server`) process,
lazily started on WS connect, idle-reaped on disconnect/timeout. The existing
`lsp_daemon` (the warm per-Lake-root compiler behind the `lean_check` tool) is **not**
touched.
- **Why:** multiplexing an interactive editor session and the agent's `lean_check` batch
  traffic over **one** Lean server means interleaving two unrelated JSON-RPC conversations
  on a single message stream — fragile, and a crash/restart of one kills the other. A
  dedicated interactive process is cleanly isolated; the agent's fast-compile path stays
  exactly as tuned (§ `lsp_daemon` restart-after, warm oleans).
- **Cost & mitigation:** each Lean+Mathlib server is hundreds of MB–GB resident. Acceptable
  for a **local single-user** tool; mitigated by lazy start (only when the user opens Live
  mode) and idle reaping. A future optimization can pool/share if memory pressure appears.
- **Rejected:** *one shared server for both.* Rejected on isolation + protocol-multiplexing
  grounds above.

## 3. Files, state, and safety

### D62 — Buffer ↔ disk/git reconciled by explicit save; editing is modal vs. agent runs
While a human edits, the **Monaco buffer** is source-of-truth (streamed to the Lean
server via `didChange` for live feedback). Persistence is **explicit**: on save (button
and/or blur) the buffer is written through the **existing** `POST /sessions/{id}/file`
→ git commit (`author=user`, `run_id=NULL`, invalidates any stale SafeVerify verdict —
already implemented, `routes/sessions.py`). When the **agent** changes the file, the new
bytes are pushed into the Monaco buffer and re-elaborated. **v1 is modal**: a session is
either in **human-edit** mode or has an **active agent run** — never both writing at once.
- **Why:** the split between "live buffer for feedback" and "committed file for the
  record" preserves **git-owns-proof-bytes** (D7/D8) and the **derived session status**
  (status comes from the latest `code_step` verdict — an uncommitted buffer must not move
  it). The modal lock is the simplest thing that makes concurrent-write conflicts
  *impossible* rather than merely rare; it can be relaxed later (D-future) once the merge
  story is designed.
- **Rejected:** *autosave every keystroke to git.* Pollutes history with hundreds of
  half-typed commits and churns the derived status. *Free concurrent edit + agent run.*
  Split-brain: whose bytes win when both write `Main.lean`? Deferred behind a real
  conflict model.
- **Consequence:** the Canvas shows a clear mode indicator + a Save affordance; entering
  Live edit is disabled while a run streams, and vice-versa.

### D63 — No Bubblewrap sandboxing
The interactive `lake serve` runs **without** a container.
- **Why:** lean4web sandboxes because it executes **untrusted code from the public
  internet**. This tool is **local, self-hosted, single-user**, and the agent **already**
  runs arbitrary Lean + `bash` via its tools (`bridge.py` gate is about *approval*, not
  isolation). A sandbox around the editor's Lean server would add operational weight
  (Bubblewrap is Linux-only; we run on macOS dev boxes too) for **zero** additional
  coverage.
- **Rejected / revisit trigger:** if a v-future ever exposes the adapter to untrusted
  remote users, this decision **must** be revisited (re-introduce Bubblewrap on Linux, or
  a container per session). Logged here so the assumption is explicit, not accidental.

### D64 — Reuse the session's existing Lake root as the "project"
`lake serve` is started with `cwd` = the session's own proof directory (loose →
`workspace/proofs/<session>/…`; project session → that project's Lake root), the **same**
root the agent's tools resolve against. No `leanweb-config.json`, no `Projects/` registry.
- **Why:** the whole point is that the human and the agent edit the **same file in the
  same environment** — imports, Mathlib pin (`lean-toolchain`), and `lakefile` must be
  identical to what the agent sees, or "hand it to the agent" breaks. Reusing the Lake root
  the session already lives in guarantees that for free and avoids inventing a parallel
  project model.
- **Consequence:** the proxy's URI rewrite maps the browser's virtual document URI onto
  the concrete file under that root (the session's main proof path, via the existing
  `_resolve_proof_path` logic).

### D65 — Drop collaboration (Yjs/WebRTC) and URL-hash sharing
Neither the Yjs signaling/`y-monaco` shared-editing nor lean4web's `code`/`codez` URL-hash
persistence is ported.
- **Why:** single-user agent tool; sessions are already persisted (git + SQLite) and
  shareable via the existing project/session export (#14). Collaboration is a large surface
  (WebRTC signaling, CRDT merge) with no current user. Keeping it out **shrinks the client
  integration** to just the editor + LSP.
- **Rejected:** *port collab "since it's there."* Cost without a consumer; revisit only if
  multi-user editing becomes a goal.

## 4. The Canvas UI

### D66 — Monaco is a new "Live" Canvas mode, not a replacement; History mode is preserved
The Canvas gains a **mode switch**: **History** (the existing Shiki snapshot stepper +
per-step verdict badges + file selector + SafeVerify — **unchanged**, #10/#11) and **Live**
(the `lean4monaco` editor + InfoView over the WS). The `contentEditable` "✎ Edit" hack is
**retired** by Live mode.
- **Why:** History mode is genuinely valuable and already built — it replays how the proof
  evolved, which a live editor can't show. Rather than throw it away, Live mode is
  *additive*: History for reviewing the agent's trajectory, Live for hands-on editing.
  This also de-risks the rollout — Live can ship behind the toggle without disturbing the
  proven path.
- **Rejected:** *replace the Shiki viewer outright with Monaco.* Loses the snapshot
  stepper + diff gutter (#10/#11) and couples every proof view to the heavy Monaco bundle.
- **Consequence:** `Canvas.tsx` grows a mode toggle; Live mode owns a Monaco container +
  the save/lock affordances (D62); History mode is code-moved, not changed.

### D67 — The InfoView (goal state) gets a dedicated panel in the Live layout
`lean4monaco` ships the Lean **InfoView** widget (goals, expected type, term info). Mount
it in a panel beside/below the editor in Live mode. Unicode abbreviation input (`\to`) and
LSP keybindings come with the package.
- **Why:** the goal state is *the* reason a human wants a live editor — "what do I still
  have to prove here" is exactly the insight the current Canvas can't give. It's the
  highest-value single feature `lean4monaco` unlocks.
- **Consequence:** Live mode is a two-pane layout (editor | InfoView); the panel reuses the
  Canvas's existing split/resize affordances where possible.

### D68 — `lean4monaco` is integrated with Vite surgery and lazy-loaded
Adopt `lean4monaco`'s required Vite setup (node polyfills, the
`@codingame/esbuild-import-meta-url-plugin`, static-copy of the language workers, and — if
features need `SharedArrayBuffer` — COOP/COEP headers), and **dynamically import** the
Monaco/editor chunk so Live mode code-splits out of the main bundle.
- **Why:** `lean4monaco` drags in Monaco + `@codingame/monaco-vscode-*` services — it is
  **not** a drop-in component, and it is **heavy**. This is where the integration time
  actually goes (mirrors the risk we already retired for Shiki in #11). Lazy-loading keeps
  the History-only user from paying for Monaco, consistent with #11's lazy Shiki chunk.
- **Rejected:** *eager-import Monaco into the main bundle.* Bloats first paint for every
  user, including those who never open Live mode.
- **Consequence:** copy lean4web's `client/vite.config` + devDeps as the reference; expect
  a chunk of build-config debugging — **Phase 1 exists specifically to retire this early.**

## 5. The extensibility payoff (Phase 3 — designed, deferred, plan-first)

Because we **own** the proxy and the buffer (D59/D60/D62), we can do things a stock
lean4web sidecar can't. These are **not** in the current cut — each needs its own design
pass and slice — but they're the reason ownership is worth it:

- **Agent-suggested edits as Monaco inline diffs / code-lens.** When the agent proposes a
  `write_file`/`edit_file`, render it as an accept/reject inline diff in the live editor
  instead of a snapshot the human re-types.
- **Shared InfoView.** Show the same goal state the human sees to the agent (and
  vice-versa) — a common ground for "explain this goal" or "why did this tactic fail here."
- **Per-line SafeVerify / lean_check gutter markers.** Surface the SafeVerify verdict and
  `lean_check` diagnostics as gutter decorations on the live buffer, not just a badge.
- **Relax the modal lock (D62-future).** A real human⇄agent merge model (e.g. agent edits
  a region the human isn't in) once the conflict story is designed.

## 7. Build plan (phased, interleaved backend + frontend)

Same rhythm as v2.1.1: one slice at a time, each ending in something **clickable on the
UI** — plan → your input → build + tests → tested → done. **You test on the UI.**

### Phase 1 · Prove the pipe — read-only live insights  *(de-risks D68 + D60)*
The highest-risk piece (the `lean4monaco` build + a working WS→Lean pipe) with **zero**
changes to the agent or persistence. Mount Monaco **read-only** on the agent's current
file; get live diagnostics + hover + goals.
- **L1 (backend)** — `lsp_proxy.py`: WS route `/api/sessions/{id}/lsp`, per-session
  `lake serve` spawn, framed JSON-RPC pump, URI rewrite. Reuse `lsp_daemon` framing. (D60/D61/D64)
- **L2 (backend)** — lifecycle: lazy start on connect, idle reap, cleanup on disconnect;
  session→Lake-root + proof-path resolution. (D61/D64)
- **LF1 (frontend)** — Vite/build integration for `lean4monaco` + lazy chunk. (D68)
- **LF2 (frontend)** — mount `LeanMonacoEditor` **read-only** in a "Live" mode toggle,
  pointed at the WS; render diagnostics + InfoView on the current file. (D66/D67)

### Phase 2 · Make it editable
- **L3 (backend)** — save flow: buffer → `POST /file` → git; modal lock vs. runs;
  agent-change → buffer push. (D62)
- **LF3 (frontend)** — enable editing; wire save (blur/explicit) → `writeSessionFile` →
  reconcile; retire `contentEditable`; mode/lock UI. (D62/D66)
- **LF4 (frontend)** — abbreviations/keybindings/polish; SafeVerify + lean_check buttons
  act on the live buffer's saved file. (D66)

### Phase 3 · Beyond lean4web — **Unplanned (plan-first)** (§5)
Agent-suggested inline diffs · shared InfoView · per-line SafeVerify gutter markers ·
relax the modal lock. Each: design pass → slices, like the factory work.
