M10 System prompt — collaborator, plan in NL before formalizing prover prompt Tested
Rewrote the interactive prompt (prover/lea/prompt.py) from an autoformalizer into a proof collaborator: lead with a natural-language sketch, propose + confirm the Lean statement, then formalize step by step narrating each tool call. Reference blocks extracted to single-source constants shared with default; hard rules intact. Hardened into a true pause (see M10b).
M10b Hard pause — lead with the math, wait for "go ahead" harness + prompt Tested
Fix for "it dives straight into write_file". An interactive text-only turn now finishes the run instead of being nudged by _NO_PROOF_ARTIFACT_MESSAGE into writing a skeleton (agent.py); the prompt makes the first reply math-only with no tools; the intent classifier routes "go ahead"/confirmations to FORMALIZE so the resume turn proves. Eval/default variant keeps the nudge.
M11 Per-write intent summary on the step card bridge + UI Tested
The narration the model writes before a write (flushed at ToolCalled) is stashed as last_intent and passed as summary= to add_code_step, shown as the step-card body and folded in (no duplicate prose bubble). Capped at 280 chars so a long NL sketch is never folded out of the thread.
M12 On deny — explain intent + invite an alternative prover prompt Tested
Reworded the deny tool-result (agent.py) so a declined tool tells the model to explain what it was about to do and ask how to proceed — not silently retry or switch steps — and wait. Added a matching "a denied action is a redirect, not a failure" clause to the collaborator prompt.
M13 Denied approval should stay visible in the thread UI Queued
Right now resolving an approval (allow/deny) clears the card and it vanishes — so there's no record you rejected something. Keep a resolved card in the thread (mockup's .approval.resolved with a "you denied this" tag), tied to that point in the timeline, instead of removing it.
M14 Scratch files shouldn't drive session status / "Proved" store + UI Queued
A run that only wrote scratch.lean (an exact? probe) showed session status ok and the NL sketch as a green "✓ Proved" card. Partly fixed in UI (FinalCard now needs a real non-scratch compiling step). Backend follow-up: store session-status derivation should ignore scratch/probe files so a session isn't "ok" without a real proof.
M15 Abandoned/paused run shouldn't lock out all new runs bridge UX Queued
A run paused on an approval (or whose SSE client navigated away) keeps holding the global active_run_lock, so every new run fails with the cryptic "Another Lea run is already active." Fixes: auto-interrupt an orphaned run when its event stream disconnects (release the lock), and/or surface the active run in the UI with a Cancel button, and/or time out _await_decision. For now a backend restart clears it.
M16 "Proved" is a milestone card keyed on run completion bridge + store + UI Built
The green "Proved" box was styling on the terminal message, so it moved/duplicated and leaked onto Q&A turns. Reworked into a milestone card keyed on the run's outcome: bridge un-flattens completed→success (proved) vs assistant→answered (chat/pause); session_detail exposes runs[] for reload; the thread groups items by run and drops one green card after a success run (red after failed/max_turns). Shows once, never on intermediate checks, persists through later chat + reload. Removed terminalMessageId/hasRealProof.
M17 Surface non-code tool activity (e.g. search_mathlib) UI Queued
When Lea ran search_mathlib there was no card — the user couldn't tell anything was happening. Tool calls that don't produce a code step (search, read, bash, lean_check on a scratch) currently only emit a status event that's invisible unless it attaches to a step card. Show inline tool-activity chips/lines in the thread (e.g. "🔍 searching Mathlib: …") for these.
M18 "Lea is thinking…" indicator after submit UX Queued
Between sending a prompt and the first assistant_delta, there's no feedback — it feels like nothing happened. Show a blinking "Lea is thinking…" indicator in the thread while a run is active and no streaming text has arrived yet, so the user knows the request reached the backend and the model is working.
M19 First-run onboarding — point users to add a key UX Queued
A brand-new user with no provider key configured has no idea where to go (and a run would just fail). When no key is set, show a small card above the composer — e.g. "Add your model preference and API key [Open settings]" — linking straight to Settings. Detect via /api/settings (api_keys[*].configured / model requirements). Pairs with M6 (Settings page for v2).
M20 Canvas edit — signal the pending edit + ask for a summary UX Queued
After editing the proof in the canvas, there's no signal that an unsent human edit is staged. Show a badge near the composer (e.g. "✎ you edited DivSix.lean") and swap the input placeholder to prompt for context — "What did you change? A short note helps Lea pick up where you left off." That note is exactly the divergence context (D12) Lea consumes on the next run, so the summary improves the handoff.
M21 SafeVerify wrongly rejected every namespaced proof prover bug Tested
SafeVerify built its target from the bare signature (root-level div_6) while the proof declares Lea.Misc.div_6 — so it reported "declaration not found in submission" and rejected valid proofs. Fixed: safeverify.namespace_context() reproduces the submission's namespace wrapper in the target (lea/interface.py), so names match. Confirmed end-to-end: the n³−n proof now returns ok. Also deleted dead adapter/app/safe_verify.py (orphaned import of the removed eval/).
