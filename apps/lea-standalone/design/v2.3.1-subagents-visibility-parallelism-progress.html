LeaUI v2.3.1 — Sub-agents: Visibility, Control & Parallelism

Live progress · pain points surfaced testing v2.3 sub-agents on a real project (Burnside Prime Degree Theorem) · follows v2.3-concurrent-runs-progress.html
v2.3 shipped sub-agents end-to-end (spawn → collate → UI, PR #28). Driving a hard project proof through them exposed a cluster of gaps that don't show on toy problems: a running child is invisible, can't be stopped or steered, leaks Lean workers, re-discovers the environment every time, and — the headline — runs strictly sequentially even though the isolation-safety groundwork for parallelism is already done. v2.3.1 closes these. The through-line: promote children from in-process blocking calls to first-class runs, and use the project Memory as the one channel that crosses the isolation boundary.

~18M tokens on the coordinator alone (a 131:1 input:output ratio — context re-sent every turn) · ~118 leaked lean --worker processes never reaped · a proof-candidate spending its whole 150-turn budget writing throwaway explore_*/test_* files (each a cold import elaboration) · children re-discovering the same Mathlib import layout ~10× · and the currently-running child absent from the UI entirely until it finished.

Design calls resolved this session

Two axes, not one. Isolation (whose context absorbs the work) ≠ speed (serial vs concurrent). Sub-agents own the isolation axis; parallelism is a separate knob. So parallel tools ≠ sub-agents: a tool result returns into the caller's context, so parallelizing tools fills the 15M-token context faster, not slower. A sub-agent is "a tool call whose intermediate work never returns to you — only the answer does."
Isolation-safety is done; execution-concurrency is not. The ContextVar per-activation scopes (v2.3 items 21/27) make concurrent children safe — the hard part. What's missing is threaded execution: spawn_subagent still drains its child inline. So parallelism is a real execution change, not a loop tweak.
Memory is the channel across the isolation boundary. A child correctly sheds the coordinator's dead-ends, but also sheds its useful environment knowledge. .lea/memory.md already exists (curated, persists across runs via git) — the right conduit. Push distilled facts in, not the raw 15M scrollback.
Steer = NOT blind live-inject. Injecting into a running child the coordinator owns is a dual-principal trap. Instead: stop → amend → re-spawn, or inject as an event the coordinator also sees (needs first-class runs).
Concurrent search = parallel tool calls, NOT sub-sub-agents. A search child fires Loogle + semantic + grep in one turn, run concurrently (E3). Per-tool nested sub-agents are the wrong tool — a single query has nothing to isolate, so wrapping it in an activation costs more tokens, and it hits the depth-1 guard anyway. Nested delegation stays architecturally supported (raise max_depth) but deferred — not needed now.
Minimal role roster: 1 supervisor + 2 leaf roles. Differentiate on the only real boundaries — read/write × know/do: scout (premise-search, read-only) and prover (proof-candidate, read-write). Decompose/sketch = the supervisor's job (not a role); disprove = prover with a negated goal. Anything sharing a toolset and objective is a prompt variation, not a new role. Don't proliferate.
Three layers of token defense (each attacks a different bloat source): isolation (sub-agents — dead-ends never enter the coordinator's context) + Memory (Phase C — facts persist across runs) + compaction (Phase G — the transcript is pruned + summarized within a run). They compose: a compaction summary feeds Memory.
Dependency order: reap workers (B1) + daemon concurrency (B3) → visible/first-class runs (D1/E1) → threaded parallel execution (E2).
Handback contract (source-confirmed). The child authors its own distilled result; the parent receives a structured outputs dict + content (OpenHands AgentDelegateObservation) / the child's final response (SDK get_agent_final_response) — not the transcript. Lea's SubagentFinished already matches; the one gap (child metrics reading as 0) is fixed by E1.
Reference implementations — read from source (cloned locally), not docs. OpenHands app ~/Desktop/code/OpenHands — condenser suite openhands/memory/condenser/impl/ (10 strategies incl. llm_summarizing, observation_masking, pipeline) + delegation handback openhands/events/observation/delegate.py. Agent SDK ~/Desktop/code/software-agent-sdk — threaded delegation openhands-tools/openhands/tools/delegate/impl.py. Cited per-item below.

B2One candidate.lean, edited in place — no scratch/explore files proverDropped
Closed — not a real issue. The premise was that scattering scratch files (find_def.lean, explore_action4.lean…) is a bug, since each new file cold-loads Mathlib (~15–30s) while only in-place edits hit the daemon's ~0.2s path. And B1 already removed the teeth: extra files are now merely slower, not fatal (no more leaked-worker swap storm). The latency residue is best handled by incentives — iterate in place, import narrowly, probe via search_mathlib/premise-search instead of throwaway compiles — not a restriction. Nothing to build.

Phase F · Search coverage & tooling

Grep can't search by type or meaning — the real reason the scout flails. Broaden the tools; the token win is coverage (right lemma in one shot), not parallelism.

F1Multi-modal search tools in premise-search (Loogle + semantic) proverPlanned
search_mathlib is grep -r over local Mathlib — text/name only; it cannot search by type or meaning (why the scout grepped Subgroup.Basic, found nothing, and spelunked). Add Loogle (type/shape/constants search) + LeanSearch/Moogle (semantic NL→lemma) alongside grep, for multi-modal coverage. Fired concurrently as parallel tool calls (E3) inside the one search child — not sub-sub-agents. Network caveat: these are HTTP APIs (same egress concern as a cloud coordinator) — plan the local-Loogle path (build the Loogle executable once against the pinned Mathlib) for air-gapped/HPC. Token win is coverage: the right lemma in one shot, fewer flailing turns.

Phase G · Context management (the token defense)

One live run cost ~18M tokens (131:1 input:output) because the transcript re-sends every turn. Compaction + a leaner coordinator turn O(N²) into bounded. Third layer alongside isolation + Memory.

G1Context compaction (condenser: prune → summarize) proverTested ✓
Landed — new lea/condenser.py runs at each turn boundary once the prior turn's real input tokens cross context_token_limit × compaction_threshold (200k×0.75=150k default; 0 disables). Prune (deterministic, no LLM) masks superseded lean_check/read_file/search_mathlib results outside a recent window with a short placeholder — copy-on-write, shape-preserving (the tool_result + id survive, so no orphaned role:"tool"). Summarize fires only if pruning didn't get back under the trigger: folds the older middle into one GOAL/PROOF_STATE/TRIED/ERRORS/LEMMAS/NEXT summary, keeping the leading goal + recent turns verbatim (tail always starts at an assistant message → no split pairs). New Compacted event → adapter context_compacted SSE → a muted "🗜 Context compacted" marker in the thread. Config knobs on LeaConfig; on by default (defends the 15M-token coordinator). Tests: tests/condenser/test_condenser.py (30 checks incl. the loop-level Compacted emit) + adapter test_run_lea_surfaces_context_compaction. Original plan below.

Adopt OpenHands' condenser suite (cloned: openhands/memory/condenser/impl/), adapted for Lean — not invented. Their Phase-1 prune is the observation_masking condenser, Phase-2 is llm_summarizing, composed by the pipeline condenser (so "prune → summarize" is their pattern). LLMSummarizingCondenser params: max_size (event budget), keep_first=1 (the head — for us the theorem goal), max_event_length=10_000 (truncate one oversized event — a giant Lean error); it keeps a rolling summary event folding prior-summary + newly-forgotten. For us: prune drops superseded lean_check error dumps + old read_file/search_mathlib outputs (the 131:1 token sink, zero LLM cost). Its summary prompt already tracks CODE_STATE/TESTS/CHANGES — map TESTS→lean_check errors, add LEMMAS_FOUND/DEAD_ENDS. Trigger ~75% (opencode). Applies to coordinator and children; reuses our _summarize_on_max_turns/_text_only_history. OpenHands reports ~50% cost cut. Feeds Memory (C).
