LeaUI v2.3.1 — Sub-agents: Visibility, Control & Parallelism

Live progress · pain points surfaced testing v2.3 sub-agents on a real project (Burnside Prime Degree Theorem) · follows v2.3-concurrent-runs-progress.html
v2.3 shipped sub-agents end-to-end (spawn → collate → UI, PR #28). Driving a hard project proof through them exposed a cluster of gaps that don't show on toy problems: a running child is invisible, can't be stopped or steered, leaks Lean workers, re-discovers the environment every time, and — the headline — runs strictly sequentially even though the isolation-safety groundwork for parallelism is already done. v2.3.1 closes these. The through-line: promote children from in-process blocking calls to first-class runs, and use the project Memory as the one channel that crosses the isolation boundary.

What one live Burnside run exposed

~18M tokens on the coordinator alone (a 131:1 input:output ratio — context re-sent every turn) · ~118 leaked lean --worker processes never reaped · a proof-candidate spending its whole 150-turn budget writing throwaway explore_*/test_* files (each a cold import elaboration) · children re-discovering the same Mathlib import layout ~10× · and the currently-running child absent from the UI entirely until it finished.

Design calls resolved this session

Two axes, not one. Isolation (whose context absorbs the work) ≠ speed (serial vs concurrent). Sub-agents own the isolation axis; parallelism is a separate knob. So parallel tools ≠ sub-agents: a tool result returns into the caller's context, so parallelizing tools fills the 15M-token context faster, not slower. A sub-agent is "a tool call whose intermediate work never returns to you — only the answer does."
Isolation-safety is done; execution-concurrency is not. The ContextVar per-activation scopes (v2.3 items 21/27) make concurrent children safe — the hard part. What's missing is threaded execution: spawn_subagent still drains its child inline. So parallelism is a real execution change, not a loop tweak.
Memory is the channel across the isolation boundary. A child correctly sheds the coordinator's dead-ends, but also sheds its useful environment knowledge. .lea/memory.md already exists (curated, persists across runs via git) — the right conduit. Push distilled facts in, not the raw 15M scrollback.
Steer = NOT blind live-inject. Injecting into a running child the coordinator owns is a dual-principal trap. Instead: stop → amend → re-spawn, or inject as an event the coordinator also sees (needs first-class runs).
Concurrent search = parallel tool calls, NOT sub-sub-agents. A search child fires Loogle + semantic + grep in one turn, run concurrently (E3). Per-tool nested sub-agents are the wrong tool — a single query has nothing to isolate, so wrapping it in an activation costs more tokens, and it hits the depth-1 guard anyway. Nested delegation stays architecturally supported (raise max_depth) but deferred — not needed now.
Minimal role roster: 1 supervisor + 2 leaf roles. Differentiate on the only real boundaries — read/write × know/do: scout (premise-search, read-only) and prover (proof-candidate, read-write). Decompose/sketch = the supervisor's job (not a role); disprove = prover with a negated goal. Anything sharing a toolset and objective is a prompt variation, not a new role. Don't proliferate.
Three layers of token defense (each attacks a different bloat source): isolation (sub-agents — dead-ends never enter the coordinator's context) + Memory (Phase C — facts persist across runs) + compaction (Phase G — the transcript is pruned + summarized within a run). They compose: a compaction summary feeds Memory.
Dependency order: reap workers (B1) + daemon concurrency (B3) → visible/first-class runs (D1/E1) → threaded parallel execution (E2).
Handback contract (source-confirmed). The child authors its own distilled result; the parent receives a structured outputs dict + content (OpenHands AgentDelegateObservation) / the child's final response (SDK get_agent_final_response) — not the transcript. Lea's SubagentFinished already matches; the one gap (child metrics reading as 0) is fixed by E1.
Reference implementations — read from source (cloned locally), not docs. OpenHands app ~/Desktop/code/OpenHands — condenser suite openhands/memory/condenser/impl/ (10 strategies incl. llm_summarizing, observation_masking, pipeline) + delegation handback openhands/events/observation/delegate.py. Agent SDK ~/Desktop/code/software-agent-sdk — threaded delegation openhands-tools/openhands/tools/delegate/impl.py. Cited per-item below.

Phase B · Resource & correctness (do first — parallelism multiplies these)

Bugs that are annoying at N=1 and catastrophic at N=many. These gate the headline.

B1Reap leaked lean --worker processes proverPlanned
P0 A live run left ~118 idle lean --worker processes. Each new-file lean_check spawns a worker that is never killed; each holds Mathlib state (hundreds of MB+). It's the real perf killer today, and it multiplies once children run in parallel. Reap per-check / on child teardown; audit lsp_daemon worker lifecycle. Precedent: the SDK's DelegateExecutor.close() explicitly conversation.close()s every sub-agent on teardown (delegate/impl.py) — child cleanup is a first-class step there, not an afterthought.
B2One candidate.lean, edited in place — no scratch/explore files proverPlanned
P0 proof-candidate scatters candidate.lean, find_def.lean, explore_action4.lean, test_fintype.lean… Every new file forces a full cold elaboration of the heavy Mathlib imports — the daemon's ~0.2s fast path only helps in-place edits of an already-open file. Harden the prompt to a single candidate.lean edited in place; forbid scratch/explore files. Directly fixes the slow-lean_check complaint.
B3LSP daemon: concurrent checks from multiple children proverPlanned
prereq for E2 Two children compiling at once hit the same per-root daemon. Verify/harden concurrent lean_check (extend test_concurrent_checks) so parallel children don't serialize or corrupt at the daemon layer. Compose with B1 (concurrency ⇒ more workers).

Phase C · Cross-agent context (the Memory channel)

Stop every child re-deriving what the coordinator already learned. The mechanism exists — wire it.

C1Coordinator actually writes durable facts to .lea/memory.md proveradapterPlanned
Write gap. The project Memory doc exists and the prompt says "maintaining this is part of your job," yet after ~90 turns of discovery it's still the empty 3-line default. Sharpen and un-bury the instruction; nudge the coordinator to record the import map, verified lemma locations, and dead ends.
C2Propagate the compact Memory into every child's context proveradapterPlanned
Read gap. Children are spawned in-process with just their task and bypass the bridge's project-context injection, so they never see .lea/memory.md. Inject the compact Memory into each child's starting context (_child_config/spawn_subagent). Subsumes the environment re-discovery — no more each child re-finding "Subgroup lives under Algebra/Group/Subgroup."
C3Seed the environment brief at project init adapterPlanned
Pre-populate .lea/memory.md (or a skill) with this Mathlib version's import conventions + key module locations, so the map exists before any agent runs. Optional once C1/C2 flow, but removes the cold-start on a fresh project.
C4Coordinator collates via read_file, not bash ls proverPlanned
The coordinator spelunks child scratch dirs with bash ls (tripping the approval gate) instead of read_file on the candidate path the sub-agent result already hands it — and re-verifies in its 15M-token context. Point it at the candidate path directly; cut the gate friction.

Phase D · Visibility & control

You are the creator and you're flying blind. A child must be seeable, stoppable, steerable — and you should be able to pick which model each role runs, from the composer.

D1Render a sub-agent the moment it's spawned (running state) adapterfrontendPlanned
P0 A child is materialized only on SubagentFinished, so a running child is invisible and the list shows only finished ones — they all look done. Emit a spawn-started event, materialize the child as running at spawn, update on finish; live running/done indicator. Without this, N concurrent children (E2) is strictly worse than one.
D2Stop a single running sub-agent adapterproverPlanned
needs D1/E1 Children run with should_stop=None (autonomous). Thread a per-child stop signal so a user can kill a runaway child without cancelling the whole coordinator run; it returns partial findings gracefully (like summarize-on-cap).
D3Steer a sub-agent — stop→amend→re-spawn (not blind inject) adapterfrontendDesign
Resolved: not blind live-injection (dual-principal — the coordinator owns the delegation and would be steered by input it never saw). Do stop → amend → re-spawn, or inject as an event the coordinator also sees. Depends on D1/E1.
D4/model — set the coordinator model from the composer frontendPlanned
Type /model in the composer to pop the existing Spotlight model-picker card above the input (search/select), then run with that coordinator model. Near-pure reuse: ModelPicker.tsx + stores/model.ts setModel (already persists to the same settings config a run reads) — it sets the same active model the chat-head chip does. The only new piece is a composer slash-command parser, shared with /compact (G3): parse /<cmd> at the input and dispatch to a small command registry. Frontend-only.
D5/sub-agent-model — set the sub-agent model from the composer frontendadapterproverPlanned
Same picker card, second target — but this one needs a new runtime channel. Today a child's model comes only from its profile YAML model: (static) or it inherits the coordinator (_child_config: profile.model or parent.model) — there is no sub-agent model in the UI/store at all. Add a persisted sub_agent_model setting (store + settings) that the adapter threads into the child config to override profile/inherit. This is the live-UI counterpart of the heterogeneous split we do for benchmarking by hand (coordinator = GPT, children = self-hosted Deepseek on vLLM, set via the YAML model: line) — it promotes that from a file edit to a picker. Reuses D4's parser + card; the real work is the override plumb through bridge/_child_config.

Phase E · Parallel execution (the headline)

Formalize independent sub-lemmas at once. The isolation-safety is paid for; the execution isn't.

E1First-class child runs on the RunBroker adapterproverDesign
P0 Promote children from in-process blocking drains to real runs with their own SSE stream + stop hook. The umbrella that makes D1/D2/D3/E2 natural instead of four separate patches. The adapter already has the broker; the prover emits child start/progress/stop events instead of absorbing them. Also fixes child-usage tracking (why sub-agent tokens read as 0): the SDK syncs each child's metrics into the parent — parent_stats.usage_to_metrics["delegate:{id}"] = sub.get_combined_metrics(), replace not add to avoid double-count on re-delegation (delegate/impl.py). A first-class child run gets its own usage row, giving us the same rollup.
E2Threaded parallel children (copy_context per child) proverDesign
P0 The actual pain. Isolation-safety (ContextVar scopes, items 21/27) makes concurrent children safe, but execution is synchronous: spawn_subagent drains its child inline. Run each child on its own thread via contextvars.copy_context().run(...) (its own scope); barrier-join multiple spawns in a turn. LLM + lean_check are blocking I/O, so threads truly overlap. Source-validated — the SDK's DelegateExecutor (openhands-tools/.../delegate/impl.py) does exactly this: a threading.Thread per task, start() all then join() (barrier), a max_children=5 cap, two-phase spawn/delegate, and per-child exceptions isolated into an errors map so a failed child never kills its siblings. Needs B1/B3 + D1/E1.
E3Parallelize the serial tool loop (independent tools) proverPlanned
The loop runs a turn's tool calls in a serial for tc in tool_calls. Run independent tools (and spawns) concurrently for latency. Orthogonal to isolation — a smaller, separate win; keep write→check pairs serial. Chosen mechanism for concurrent multi-modal search (F1): a search child fires Loogle + semantic + grep in one turn — flat parallel calls, not sub-sub-agents.

Phase F · Search coverage & tooling

Grep can't search by type or meaning — the real reason the scout flails. Broaden the tools; the token win is coverage (right lemma in one shot), not parallelism.

F1Multi-modal search tools in premise-search (Loogle + semantic) proverPlanned
search_mathlib is grep -r over local Mathlib — text/name only; it cannot search by type or meaning (why the scout grepped Subgroup.Basic, found nothing, and spelunked). Add Loogle (type/shape/constants search) + LeanSearch/Moogle (semantic NL→lemma) alongside grep, for multi-modal coverage. Fired concurrently as parallel tool calls (E3) inside the one search child — not sub-sub-agents. Network caveat: these are HTTP APIs (same egress concern as a cloud coordinator) — plan the local-Loogle path (build the Loogle executable once against the pinned Mathlib) for air-gapped/HPC. Token win is coverage: the right lemma in one shot, fewer flailing turns.

Phase G · Context management (the token defense)

One live run cost ~18M tokens (131:1 input:output) because the transcript re-sends every turn. Compaction + a leaner coordinator turn O(N²) into bounded. Third layer alongside isolation + Memory.

G1Context compaction (condenser: prune → summarize) proverPlanned
Adopt OpenHands' condenser suite (cloned: openhands/memory/condenser/impl/), adapted for Lean — not invented. Their Phase-1 prune is the observation_masking condenser, Phase-2 is llm_summarizing, composed by the pipeline condenser (so "prune → summarize" is their pattern). LLMSummarizingCondenser params: max_size (event budget), keep_first=1 (the head — for us the theorem goal), max_event_length=10_000 (truncate one oversized event — a giant Lean error); it keeps a rolling summary event folding prior-summary + newly-forgotten. For us: prune drops superseded lean_check error dumps + old read_file/search_mathlib outputs (the 131:1 token sink, zero LLM cost). Its summary prompt already tracks CODE_STATE/TESTS/CHANGES — map TESTS→lean_check errors, add LEMMAS_FOUND/DEAD_ENDS. Trigger ~75% (opencode). Applies to coordinator and children; reuses our _summarize_on_max_turns/_text_only_history. OpenHands reports ~50% cost cut. Feeds Memory (C).
G2Orchestrator-first coordinator (delegate proving, don't grind) proverPlanned
The coordinator both delegates and grinds proofs itself (it wrote Burnside.lean ~30× in its own context) — the structural cause of the 15M-token coordinator. Retune it to plan → delegate → collate → promote → maintain memory, pushing the write→check→error grind into prover children. A role-boundary fix, not a new agent — and the highest-leverage token lever.
G3/compact — user-triggered manual compaction frontendadapterPlanned
A slash command (like Claude Code's /compact) so the user can compact on demand, not only at the automatic threshold (G1) — before a long follow-up, or when they watch the context bloating. Same condenser under the hood (pipeline([observation_masking, llm_summarizing])); the command just fires it now instead of waiting. Frontend: a /compact command in the composer; adapter: an endpoint that runs the condenser on the active session, persists the condensed state, and streams a "freed ~N tokens" note. G1 is the auto trigger, G3 the manual one — one mechanism, two entry points.
