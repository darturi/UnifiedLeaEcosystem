# v2.3.1 ‚Äî Sub-agents: Visibility, Control & Parallelism (Architecture)

**Status:** design / first-principles plan. No code yet.
**Backlog tracker:** `v2.3.1-subagents-visibility-parallelism-progress.html` (items A‚ÄìE).
**Predecessor:** v2.3 shipped sub-agents (spawn ‚Üí collate ‚Üí UI, PR #28). This doc is the
architecture for closing the gaps that live-testing a hard project proof exposed.

---

## 0. TL;DR ‚Äî the one inversion

Everything in v2.3.1 follows from a single architectural change:

> **A child stops being an in-process blocking function that returns a string, and
> becomes a first-class *run* ‚Äî independently scheduled, observable, and controllable.**

Today `spawn_subagent` calls `_run_child`, which drains the child's `run_events`
generator to completion *on the caller's thread*, absorbs its events, and returns a
distilled string. That one choice is the root cause of **all** the symptoms:

| Symptom (from live testing) | Caused by |
|---|---|
| Running child invisible in the UI until it finishes | events absorbed, not streamed |
| Can't stop a runaway child without killing the whole run | no per-child control handle |
| Can't steer a child | child isn't addressable while alive |
| "Spawned several" but they ran one-at-a-time | synchronous inline drain |
| ~118 leaked `lean --worker` procs | no per-child resource lifecycle |

Fix the seam and every symptom becomes tractable. The rest of this doc is *how*, chosen
from first principles and cross-checked against opencode and OpenHands.

---

## 1. Reference architectures ‚Äî what we borrow, what we reject

### OpenHands (the closer match)
- **In-process, thread-based execution.** `DelegateTool` spawns sub-agents as concurrent
  `threading.Thread`s (`DelegateExecutor.spawn` then `.delegate`, capped `max_children=5`);
  `TaskToolSet` runs them **sequentially, blocking** the parent. ‚Üí *Validates our plan:
  threads for concurrency, a cap, and a spawn/delegate split.*
- **Typed event stream + delegation visualizer.** Sub-agents are isolated by
  `conversation_id`, tracked by a `TaskManager` (status, metadata, results), and rendered
  through a delegation visualizer. ‚Üí *This is exactly "child = first-class run on a
  stream," which is our target.*
- **`ConfirmationHandler`** ‚Äî a child can reach `WAITING_FOR_CONFIRMATION` and block for
  approval. ‚Üí *We keep children autonomous, but note the hook exists once a child is a
  first-class run (future per-child approval, cheap to add).*
- **`_agent_factories` registry**, 5-tier precedence (programmatic > project/user file >
  plugin > built-in). ‚Üí *Our YAML profiles in `lea/agents/` already are this.*

### opencode (the cautionary match)
- Subagents are **child sessions with their own lifecycle**; per-agent config is
  `model` / `permission` (per-tool) / `steps` (max iterations) / `prompt`. ‚Üí *This is our
  `AgentProfile` (model / tools / max_turns / system_prompt) ‚Äî we already match it.*
- **Their open bugs are our exact targets:** parent "stuck loading" while a subagent is
  blocked with no visibility/cancel ([#10802]), no force-kill API ([#21176]), subagents
  hang with no timeout ([#13841], [#22167]). ‚Üí *We are not behind a polished opencode; we
  must simply not ship their gaps.*

### Verdict
Our target **is** OpenHands' posture ‚Äî event-stream-native, thread-based concurrent
children, a task-manager tracking running children ‚Äî implemented on the infrastructure we
already own (**the adapter `RunBroker` + SSE**, and **opencode-style YAML profiles**). We
reject: a separate prover HTTP server (breaks the in-process library seam), async-rewriting
the prover (unnecessary ‚Äî the work is I/O-bound), and absorbing child events (the root cause).

### Source-grounded (local clones, read directly ‚Äî not docs)
Both are cloned; the claims above are verified against source (paths in the
`openhands-reference-clones` memory):
- **Threaded delegation** ‚Äî SDK `openhands-tools/openhands/tools/delegate/impl.py`:
  `DelegateExecutor` runs a `threading.Thread` per task, `start()`-all then `join()`-all
  (barrier), `max_children=5`, two-phase `spawn`/`delegate`, per-child exceptions isolated,
  and `close()` closes each sub-agent on teardown (our worker-reap precedent ‚Üí B1).
- **Child metrics ‚Üí parent** (same file): `parent_stats.usage_to_metrics["delegate:{id}"] =
  sub.get_combined_metrics()`, *replace* not add. Fixes Lea's "child tokens read as 0" gap;
  a first-class child run (E1) gives us the same rollup.
- **Handback** ‚Äî app `openhands/events/observation/delegate.py`:
  `AgentDelegateObservation(outputs: dict, content: str)` ‚Äî structured result, not transcript.
  Lea's `SubagentFinished` matches.
- **Compaction** ‚Äî app `openhands/memory/condenser/impl/`: 10 strategies; our G1 =
  `pipeline([observation_masking, llm_summarizing])` (prune ‚Üí summarize). See ¬ß7.

## 2. The seam ‚Äî who owns what (modularity)

lives (the adapter).** The prover must not learn about SSE, the DB, or the broker.

The **only** new prover‚Üíadapter contract is an **event sink** (a thread-safe callback the
adapter passes into `run_events`) plus **typed child events**. Everything downstream
(broker, SSE, DB, UI) already exists.

### Role roster (minimal ‚Äî 1 supervisor + 2 leaf roles)
Differentiate on the only real capability boundaries ‚Äî *read/write* √ó *know/do* ‚Äî not on prompt
flavor:
- **Supervisor / coordinator** (not a sub-agent): plan ‚Üí decompose ‚Üí delegate ‚Üí collate ‚Üí promote ‚Üí
  maintain Memory. *Orchestrator-first* ‚Äî it delegates the proving grind rather than doing it, so its
  own context stays lean (the structural fix for the 15M-token coordinator, G2). Decompose/sketch
  lives here, not in a role.
- **`scout`** (`premise-search`, read-only) ‚Äî acquire knowledge. Tooling upgrade (F1): grep
  (`search_mathlib`) **+ Loogle** (type/shape search) **+ LeanSearch/Moogle** (semantic), fired as
  *parallel tool calls* in one child (E3), **not** sub-sub-agents. Plan a local-Loogle build for
  air-gapped runs.
- **`prover`** (`proof-candidate`, read-write) ‚Äî produce one verified sub-proof. Disprove = the same
  role with a negated goal (no separate role).

Anything sharing a toolset *and* objective is a prompt variation expressed in the task text ‚Äî **not**
a new role. (opencode ships 3 subagents; we ship 2 + a supervisor.)

## 3. The event model (typed, extensible)

A child **is** a run, so it should speak the **same event vocabulary** as a top-level run ‚Äî
just tagged with its `child_id`. This is the elegant unification: we don't invent a parallel
"subagent event" language; we reuse `TurnStarted`, `ToolCalled`, `CheckResult`,
`AssistantTextDelta`, `UsageUpdated`, `Finished`, each carrying an optional `run_id`/`child_id`.

New/changed types:
- **`SubagentStarted(child_id, parent_id, role, task, depth, model)`** ‚Äî emitted *before* the
  child's first turn. This is the event that fixes invisibility (item D1).
- **Existing typed events, tagged with `child_id`** ‚Äî the child's turns/tools/checks stream
  live instead of being absorbed.
- **`SubagentFinished(...)`** ‚Äî unchanged in spirit; still carries the distilled result +
  transcript for collation (item 25) and the coordinator's tool_result.

Extensibility: a new lifecycle signal (e.g. `SubagentPaused`) is one dataclass + one
supervisor branch. No schema migration ‚Äî child runs reuse the `runs`/`timeline` tables.

### Event transport (the one subtle mechanism)
The coordinator's own events are *pulled* (the adapter iterates its generator). But children
run on **other threads concurrently**, and you can't yield into a single-threaded generator
from another thread. So:

> The adapter passes a **thread-safe `event_sink` callback** into `run_events`. **Child
> threads push** their events to the sink; the coordinator keeps yielding as today. The sink
> fans events onto the right broker stream by `child_id`.

This is a small, well-contained interface change (add an optional `event_sink=` param). It's
also the natural future path to unify everything on push, but v1 doesn't require that.

## 4. Execution model (parallelism ‚Äî item E2)

- Each `spawn_subagent` dispatches its child via `contextvars.copy_context().run(child_loop)`
  on a worker thread. Each copy already carries its own scope (working_dir, run_key, depth,
  registry overlay, results collector ‚Äî all made ContextVar-based in v2.3 items 21/27). **This
  is why concurrent children are safe today; only the execution is missing.**
- A coordinator turn with N `spawn_subagent` calls: **dispatch all N concurrently, then
  barrier-join** before the turn continues (OpenHands `DelegateTool` model). Other tools in the
  turn stay serial (never parallelize a write‚Üícheck pair).
- **Concurrency cap:** a `max_children` semaphore (start at **4**, matching our existing run
  cap and OpenHands' 5). Excess spawns queue.
- **Why threads, not async:** `litellm.completion` and `lean_check` are blocking **I/O** ‚Äî the
  GIL releases during both, so threads genuinely overlap. Async would require rewriting the
  entire synchronous prover loop for zero additional benefit on I/O-bound work. (OpenHands, also
  Python, chose threads for the same reason.)
- **Two-phase spawn/delegate (OpenHands):** optional future extension (spawn a pool, then feed
  tasks). Not needed for v1; the architecture doesn't preclude it.

## 5. Control plane (items D2/D3)

- **Stop a child (D2):** the prover exposes a per-child `should_stop` hook (children currently
  pass `None`). The adapter, on a UI stop request for a child run, flips that hook via the
  broker (keyed by child run id). The child ends gracefully at the next turn boundary and
  returns **partial findings** (reuse the summarize-on-cap path ‚Äî a capped/stopped child still
  hands back what it learned).
- **Steer a child (D3):** **not** blind live-injection (dual-principal: the coordinator owns the
  delegation and would be steered by input it never saw). Two sanctioned forms:
  1. **stop ‚Üí amend ‚Üí re-spawn** ‚Äî the UI stops the child and re-delegates the same task with the
     user's added note appended. Each child stays a clean single-prompt activation.
  2. **inject-as-visible-event** ‚Äî a user message enters the child stream *and* is surfaced to the
     coordinator's context, so no principal is blind. Deferred; requires the event model (¬ß3).
- **Per-child approval (future):** children stay autonomous by default, but because a child is now
  a first-class run, the approval-gate hook that top-level runs use is *available* to wire per role
  (mirrors OpenHands' `ConfirmationHandler`). Extensibility, not v1 scope.

## 6. Resource lifecycle (items B1‚ÄìB3 ‚Äî correctness, do first)

Concurrency multiplies leaks, so these gate everything else.
- **B1 ‚Äî reap `lean --worker`:** tie LSP-session lifecycle to child teardown. When a child run
  ends, its opened LSP documents are closed and its workers reaped. (~118 leaked in one live run;
  under N-way concurrency this is fatal.)
- **B2 ‚Äî one `candidate.lean`, edited in place:** the LSP daemon's ~0.2s fast path only applies to
  *in-place edits of an already-open file*. proof-candidate scattering `explore_*`/`test_*` files
  pays a full cold import elaboration per file. Harden the role prompt to a single candidate edited
  in place; forbid scratch files.
- **B3 ‚Äî daemon concurrency:** N children share the per-root daemon; verify/harden concurrent
  `didOpen`/check (extend `test_concurrent_checks`). Prerequisite for E2.

## 7. Context management ‚Äî the three-layer token defense

One live run cost **~18M tokens** on the coordinator alone (131:1 input:output) because the
transcript re-sends every turn. Three *complementary* layers attack three different bloat sources;
together they turn O(N¬≤) into bounded.

**Layer 1 ‚Äî Isolation** (the sub-agent feature itself): dead-end exploration happens in a child
context and never enters the coordinator's.

**Layer 2 ‚Äî Memory** (`.lea/memory.md`, items C1‚ÄìC4): durable, distilled facts that persist *across
runs*. The mechanism exists (per project, git-persisted, "both you and Lea append") but is unwired on
both ends:
- **C1 (write):** the coordinator isn't populating it (empty default after ~90 turns). Sharpen and
  un-bury the "maintain memory" instruction; record the import map, verified lemma locations, dead ends.
- **C2 (read):** children spawn with *only their task* and bypass the bridge's project-context
  injection. Inject the **compact** Memory into each child's context. Subsumes environment
  re-discovery (each child re-finding "`Subgroup` lives under `Algebra/Group/Subgroup`").
- **C3:** seed `.lea/memory.md` with the Mathlib-version import brief at project init.
- **C4:** coordinator collates via `read_file` on the candidate path the result already gives it, not
  `bash ls` (which trips the gate and re-inflates its context).

Memory is the *controlled boundary crossing* ‚Äî curated, compact, the one channel that crosses isolation.

**Layer 3 ‚Äî Compaction** (items G1‚ÄìG3): prune + summarize the transcript *within* a run. Ported from
OpenHands' condenser suite (`openhands/memory/condenser/impl/`), not invented:
- **G1 ‚Äî automatic condenser** = `pipeline([observation_masking, llm_summarizing])`. **Prune**
  (observation-masking) drops superseded `lean_check` error dumps + old `read_file`/`search_mathlib`
  outputs ‚Äî the dominant token sink, reclaimed with zero LLM cost. **Summarize** (llm-summarizing) folds
  the pruned middle into a rolling summary. `keep_first` = the **theorem goal** (never summarized away);
  keep recent K turns + current proof state verbatim; trigger ~75% of context (opencode). Their summary
  prompt already tracks `CODE_STATE`/`TESTS`/`CHANGES` ‚Äî map TESTS ‚Üí `lean_check` errors, add
  LEMMAS_FOUND/DEAD_ENDS. Applies to coordinator *and* children; reuses our
  `_summarize_on_max_turns`/`_text_only_history`. OpenHands reports ~50% cost cut.
- **G2 ‚Äî orchestrator-first coordinator:** delegate the proving grind instead of doing it (role roster,
  ¬ß2). A role-boundary fix ‚Äî the highest-leverage token lever.
- **G3 ‚Äî `/compact` (manual):** a slash command (like Claude Code's) so the user fires the *same*
  condenser on demand, not only at the auto threshold. Frontend command + adapter endpoint that runs the
  condenser on the active session and reports "freed ~N tokens." G1 is the auto trigger, G3 the manual
  one ‚Äî one mechanism, two entry points.

The layers compose: **a compaction summary is a natural feed into Memory** ‚Äî durable facts extracted
during a prune-summarize cross into `.lea/memory.md`.

## 8. Rollout ‚Äî dependency-ordered, each slice shippable

1. **Correctness (B1, B2, B3)** ‚Äî no UX change; makes concurrency survivable. *Ship first.*
2. **Event model + prover emits child lifecycle events** (¬ß3) ‚Äî internal; behavior unchanged, but
   the sink + `SubagentStarted` are now in place.
3. **Adapter `SubagentSupervisor` + D1 render-on-spawn** ‚Äî children appear live (visibility). This
   alone resolves most of the "I'm flying blind" pain.
4. **D2 stop** ‚Äî control over a running child.
5. **E2 threaded parallel children** ‚Äî the payoff (parallel sub-lemmas), now *safe* (B done) and
   *visible* (D1 done).
6. **Context: Memory (C1‚ÄìC4) + Compaction (G1 auto, G3 `/compact`)** ‚Äî independent track; can land in
   parallel with 1‚Äì5. G2 (orchestrator-first coordinator) pairs with the role roster (¬ß2).
7. **Search tooling (F1)** ‚Äî multi-modal `scout` (Loogle + semantic), an independent scout upgrade.
8. **Extensibility follow-ons:** D3 steer, E3 tool-loop parallelism, E1 two-phase spawn/delegate,
   per-child approval.

Each numbered step is independently testable and independently valuable ‚Äî no big-bang.

## 9. What this buys us (modularity / extensibility payoffs)

- **New role** = a new YAML file (opencode-style; already works).
- **New child event** = one dataclass + one supervisor branch; no migration (child runs reuse
  `runs`/`timeline`).
- **Heterogeneous models** (your GPT-coordinator + Deepseek-children) = the profile `model` field
  (already exists); first-class runs make each child's model visible per run in the UI.
- **New control** (pause/resume, per-child approval) = the first-class-run hooks already exist.
- **Nested delegation** = the same machinery; raise `max_depth` (keep the guard) ‚Äî no new code path.
- **The FQB/eval harness** can drive the same first-class-run model headlessly (children as runs, not
  hidden calls), which is what makes a benchmarked heterogeneous coordinator/child setup observable.

## 10. Explicitly rejected

- **Blind live-injection into a running child** ‚Äî dual-principal; use stop‚Üíamend‚Üíre-spawn or
  visible-event injection.
- **Parallel tools as a substitute for sub-agents** ‚Äî wrong axis: a tool result returns into the
  caller's context (fills the 15M budget *faster*); sub-agents distill. Parallelizing the tool loop
  (E3) is a separate, smaller latency win.
- **Absorbing child events in-process** ‚Äî the current anti-pattern and the root cause.
- **A separate prover HTTP service for children** ‚Äî violates the in-process library seam; children
  are threads, not services.
- **Async-rewriting the prover loop** ‚Äî unnecessary for I/O-bound work; threads suffice.
