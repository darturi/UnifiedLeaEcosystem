## 1. Vision

v2 is a **three-region** app — Sessions · Chat · Canvas (`v2-ui-design.md` §1). v2.1 adds
one new top-level surface: the **project window**, which *replaces* the chat+canvas pair
when a project is open (the sidebar stays).

```
┌──────────┬──────────────────────────────────────────────┐
│ SIDEBAR  │   PROJECT WINDOW  (when a project is open)     │
│          │  ┌────────────────────────┬─────────────────┐ │
│ Projects │  │  Overview · Blueprint  │   right rail:    │ │
│  · Epsi… │  │  · Filesystem  (tabs)  │  Instructions    │ │
│          │  │                        │  Memory · Files  │ │
│ Chats    │  │  graph / sessions /    │                  │ │
│  (loose) │  │  files / md editors    │                  │ │
└──────────┴──┴────────────────────────┴─────────────────┴─┘
```

A project window is where a mathematician **sees the whole proof effort at once**: the
lemma **dependency graph** (what's planned, what's proved, how it chains), the project's
sessions, the shared files, and the three living markdown docs the agent and human
co-author. Opening a session inside a project drops back into the familiar Chat+Canvas.

The motivating loop (search → tex/blueprint → Lean verification, decomposing into
chaining lemmas) maps onto the window: **Blueprint** tab = the decomposition + graph;
**Overview** = run sessions that prove the nodes; **Filesystem** = the shared artifact you
share with collaborators.

## 1a. Foundational principles

Carry v2's two principles (P1 collaborator-not-autoformalizer, P2 writeable canvas) and
add three project-scoped ones.

### P3 — The project window is *tabbed*, not a fourth pane
A project has more facets than fit one screen (sessions, graph, files, three docs). Rather
than cram them or add a fourth resize-pane, the window is **tabbed**: **Overview** (hero +
compose + sessions + rail), **Blueprint** (graph + markdown authoring), **Filesystem**
(the repo tree). The per-session Chat+Canvas is unchanged and untouched (backend D33: the
canvas stays Lean-only).

### P4 — The graph is *navigation*, not just a picture
The dependency graph isn't a static diagram — every node links back into the work.
Clicking a node opens the sessions that formalized it (backend D29 derives node↔session),
and its status color is the *live* Lean verdict (backend D28). The graph is the project's
table of contents.

### P5 — Three docs, one editor
Instructions, Memory, and Blueprint are three uniform `.lea/*.md` files (backend
D25/D26/D28). The UI treats them uniformly: **one** reusable markdown view+edit component,
parameterized by which doc it edits. Memory and Blueprint are *also agent-written*, so the
UI refreshes them when a run touches them (backend D33's "blueprint updated" signal).

## 2. Information architecture

Extends `v2-ui-design.md` §2 (which already decided the shape; v2 left it as a placeholder):

- **Sidebar has two groups** (now both built): **Projects** (each opens the project window)
  and **Chats** (loose, non-project sessions only).
- **Project sessions never appear in the Chats list** — they live in the project window's
  Overview tab. The sidebar stays uncluttered.
- **Search is global (⌘K)** — the *only* way to reach a project session from outside its
  window, so it spans loose chats + project sessions (each tagged with its project).
- **A project window has three tabs:** Overview · Blueprint · Filesystem. A breadcrumb
  (`Projects / Epsilon`) returns to the loose chat view.

Mental model unchanged: a session belongs to **at most one** project, or none.

## 3. Project window — layout & decisions

### D35 — A `projects` Zustand store + `useProjectGraph` hook (no prop-drilling)
Project state lives in a new `stores/projects.ts` (the open project, its detail, sessions,
the three docs' content, files, the derived graph) — mirroring v2's `proofSession`/
`sessions` stores. Graph polling/refresh is a `hooks/useProjectGraph.ts`. Components read
the store; nothing is prop-drilled. (Follows the v2 frontend architecture memory.)
- **Why:** v2 already standardized "shared state = a store, side-effects = a hook." A
  project is shared across the window, three tabs, and the rail — exactly what a store is
  for. Keeps `App.tsx` thin (it routes loose-chat vs project-window; it doesn't hold
  project state).
- **Rejected:** threading the open project through props (the v1 prop-drilling the v2
  refactor explicitly removed).

### D36 — Routing: the project window replaces chat+canvas; `App.tsx` switches on view
`App.tsx` gains a top-level view selector: **loose** (current Sidebar+Chat+Canvas) vs
**project** (Sidebar + ProjectWindow). Selecting a project in the sidebar sets the open
project and switches the view; the breadcrumb / picking a loose chat switches back. This is
the v2 mockup's `showChat()`/project-view toggle, done in React state.
- **Why:** the project window is a different workspace, not a panel inside the chat. One
  clean switch keeps both layouts simple (P3).

### D37 — Project window = hero + tabbed body + right rail (extends the mockup)
The **Overview** tab matches the mockup: hero (`∑ Epsilon · namespace Lea.Epsilon` +
description) → "new proof in this project" composer → the project's sessions list (title +
fully-qualified lemma name + status dot + time). The **right rail** holds the three cards —
📋 Instructions, 🧠 Memory, 📁 Files — each opening its editor/manager. **Blueprint** and
**Filesystem** are sibling tabs (see §4, §6).
- Creating a session from the in-project composer → `POST /api/projects/{id}/sessions`
  (backend D23, working dir = project repo); opening a session row loads the normal
  Chat+Canvas, scoped to the project.
- **Why match the mockup:** it was the agreed v2.1 placeholder; we're realizing it, plus
  promoting Blueprint/Filesystem to tabs (they didn't exist in the mockup rail beyond a
  `blueprint.md` file row).

### D38 — The dependency graph (the centerpiece) — React Flow + dagre
The **Blueprint** tab's graph renders `GET /api/projects/{id}/graph` →
`{nodes(kind,status,sessions,last_modified_by), edges}` with **React Flow (`@xyflow/react`)**
laid out by **dagre** (top-down DAG). **Shape = kind** (box = definition, ellipse =
lemma/theorem); **color = status** (white planned · blue stated/ready · green proved · red
failed) — modeled on Lean's `leanblueprint` dep-graph. Clicking a node opens its attributed
sessions (P4 / backend D29). Pan/zoom/fit built in.
- **Why React Flow + dagre:** the mainstream, maintained React graph lib; custom node
  components give us box-vs-ellipse + status colors trivially; dagre auto-lays-out a
  dependency DAG so we never hand-place nodes. Far less code than raw SVG/d3, and it's the
  one new frontend dependency (logged).
- **Rejected:** hand-rolled SVG + manual layout (reinventing dagre); cytoscape/d3-force
  (force layout is wrong for a DAG — we want layered top-down); embedding leanblueprint's
  graphviz output (a static image, not clickable — breaks P4).
- **Status legend + a markdown authoring sub-view** share the tab (toggle graph ⇄ raw
  `.lea/blueprint.md`), reusing the §5 editor.

### D39 — One `MarkdownDoc` component for the three `.lea/*.md` docs (P5)
A reusable `MarkdownDoc` (rendered view via the existing `markdownParser`/`mathRenderer`,
plus an edit mode → textarea → PUT) backs Instructions, Memory, and the Blueprint authoring
view. Parameterized by `{ get, put, title, agentWritten }`.
- **Instructions** (`/instructions`, user-authored): the rail card's editor.
- **Memory** (`/memory`, agent+human): rail card editor; `agentWritten` → refresh on the
  run's "blueprint updated"/asset signal so agent edits appear live.
- **Blueprint** (`/blueprint`, agent+human): the authoring sub-view of the Blueprint tab,
  beside the graph.
- **Why one component:** backend made them uniform files; the UI should too. Avoids three
  near-identical editors.

### D40 — Files card + Filesystem tab share an upload/download/edit core
The rail **Files** card is the quick view (list + upload + download); the **Filesystem**
tab (§6) is the full repo tree with in-place editing. Both hit the same endpoints
(`/files`, `/file`, `/tree`, `/export`). Upload is a drag-drop / file-input →
`multipart POST /files` (backend D27).
- **Why:** the card and the tab are two views of the same project repo; one client core,
  two surfaces.

### D41 — ⌘K global search overlay
Realize the mockup's overlay: a centered modal, sectioned results (**Loose chats** /
**Inside projects**, project-tagged), keyboard-driven (⌘K open, ↑/↓, Enter, Esc), backed by
`GET /api/search?q=`. It's the only path to a hidden project session.
- **Why:** IA §2 hides project sessions from the sidebar, so search is load-bearing, not a
  nicety.

### D42 — Live refresh during a project run
While a project session runs, the graph/blueprint/memory can change (the agent writes
`.lea/*.md`). The runner emits the light **"blueprint updated"** asset signal (backend
D33); `useProjectGraph` listens (the run SSE is already open for the active session) and
re-fetches `/graph` + the affected doc. The project sessions list refreshes on run
completion.
- **Why:** the graph is the agent's task board; it should move as the agent works, not only
  on manual reload.
- **Simplest version first:** if wiring the asset signal into the project window proves
  fiddly while a session (not the project window) owns the SSE, fall back to a poll on
  `useProjectGraph` (e.g. while any project run is active) — logged as the cheap path.

### D43 — `github_token` in the Settings page (redacted)
Settings gains a **GitHub token** field (for D34 push), shown like the provider keys:
presence only ("configured"), never the value, write-only via `PUT /api/settings`. The
per-project remote URL is set in the Filesystem tab, not Settings.
- **Why:** mirrors v2's provider-key handling exactly; the token is global, the remote is
  per-project.
- **Security:** never render the token back; never log it; push is an explicit button with
  a target confirm (backend D34).

## 4. The dependency graph (detail)

The visualization the user asked for ("so users understand what the agent is trying to
accomplish"). Modeled on `leanblueprint`'s dep-graph but **interactive**.

- **Data:** `GET /api/projects/{id}/graph` → nodes + edges. Each node:
  `{ key, kind, lean, status, sessions[], last_modified_by, statement }`.
- **Layout:** dagre, rank top→bottom, edges = `uses` (dependency → dependent).
- **Node shape (kind):** definition → **box**; lemma/theorem → **ellipse/rounded**.
- **Node color (status, derived live — backend D28):**
  white `planned` · blue `stated`/`ready` · green `proved` · red `failed`. A small legend
  sits in the tab.
- **Interactions (P4):** hover → statement tooltip; **click → side panel** with the
  statement, the Lean decl name, status, and the **attributed sessions** (open one →
  Chat+Canvas). "Open `.lea/blueprint.md`" toggles the authoring sub-view (§5).
- **Empty / nascent project:** no nodes yet → an empty-state nudge ("Describe the
  decomposition in the Blueprint, or start a proof — nodes appear as lemmas are planned and
  proved").
- **Live:** D42 refresh.

## 5. The three markdown docs (Instructions / Memory / Blueprint)

All three are `.lea/*.md` (backend D25/D26/D28), edited through the one `MarkdownDoc`
component (D39):

| Doc | Endpoint | Author | UI home |
|---|---|---|---|
| Instructions | `/instructions` | user | Overview rail card |
| Memory | `/memory` | agent + human | Overview rail card (live) |
| Blueprint | `/blueprint` | agent + human | Blueprint tab authoring sub-view (live) |

- **View mode:** rendered markdown (KaTeX math via the existing `mathRenderer`).
- **Edit mode:** raw textarea → PUT (write+commit on the backend). A subtle "edited by
  agent" timestamp when `last_modified_by` is the agent.
- **Memory shape:** a bulleted list is the natural form (the mockup's `▪` items), but it's
  free markdown — no enforced schema (backend D26).

## 6. Filesystem tab (D40)

Exposes the project git repo (backend D34) — "see everything, edit, download, share."

- **Tree:** `GET /api/projects/{id}/tree` → the repo's files (proofs under `Lea/<Project>/`,
  `.lea/` assets, uploads). **Hide** `.git/`, `.lake/`, oleans (backend Open Q-7 resolved:
  source + assets only).
- **Viewer/editor:** click a file → `GET /file?path=` → view; **edit** → `PUT /file`
  (write+commit, path-guarded). A `.lean` edit re-uses the standalone `check` for a verdict
  (like the v2 canvas); editing `.lea/blueprint.md` refreshes the graph (D42).
- **Upload:** drag-drop / picker → `multipart POST /files` (shared with the Files card).
- **Download:** a button → `GET /export` (zip / git bundle).
- **Share to GitHub:** set the remote (`PUT /git/remote`) inline, then **Push** (`POST
  /git/push`) — an **explicit button** with a "push to `<remote>`?" confirm (outward-facing;
  backend D34). Disabled with a hint if no `github_token` is configured (links to Settings).

## 7. Visual language additions

Carry v2's tokens (`v2-ui-design.md` §5). New:

- **Graph node colors** reuse the semantic palette: proved = green `#4f8a5b`, failed = red
  `#c0564a`, stated/ready = blue `#4a6fa5`, planned = neutral panel/line. Box vs ellipse for
  kind.
- **Tabs:** a light underline-style tab strip in the project window header (Overview ·
  Blueprint · Filesystem), terracotta accent on the active tab.
- **Rail cards** reuse the mockup's `.rail-card` styling (already in the v2 CSS as the
  placeholder project window).

## 8. State & wiring (React + TypeScript)

Maps onto the v2 frontend architecture (Zustand stores + hooks + `lib/types.ts` +
`lib/api.ts`). **Bold = new.**

### 8.1 New store + hook
- **`stores/projects.ts`** — open project id, project detail (meta + sessions + files),
  the three docs' content, the derived graph, loading flags; actions: `openProject`,
  `createProject`, `refreshGraph`, `saveDoc(kind, md)`, `uploadFile`, `deleteFile`,
  `setRemote`, `push`.
- **`hooks/useProjectGraph.ts`** — fetch + live-refresh the graph (D42).

### 8.2 New API surface (`lib/api.ts` + `lib/types.ts`)
Typed fetchers for every D31/D34 route: `listProjects`, `createProject`, `getProject`,
`updateProject`, `deleteProject`, `createProjectSession`, `getDoc`/`putDoc` (instructions /
memory / blueprint), `getGraph`, `getTree`, `getFile`/`putFile`, `uploadFile` (FormData),
`downloadFile`, `exportProject`, `setRemote`, `pushProject`, `search`. New types:
`Project`, `ProjectDetail`, `GraphNode`, `GraphEdge`, `ProjectFile`, `TreeEntry`,
`SearchResult`.

### 8.3 New components
- `ProjectWindow.tsx` (router for the tabs + hero + rail)
- `ProjectGraph.tsx` (React Flow + dagre; custom node = `GraphNode.tsx`)
- `MarkdownDoc.tsx` (the one editor, D39)
- `FilesystemTab.tsx` (tree + viewer/editor + upload + export + push)
- `SearchOverlay.tsx` (⌘K, D41)
- Sidebar: add the **Projects group** to the existing `Sidebar.tsx`.
- Settings: add the **GitHub token** field to `SettingsPage.tsx` (D43).

### 8.4 Reuse (no change)
- The whole Chat+Canvas + run SSE stack (`useProofStream`, `ChatThread`, `Canvas`,
  `timeline.mjs`) — a project session is just a session with a `project_id`; the run flow
  is identical (backend D31 reuses the run SSE unchanged).
- `markdownParser` / `mathRenderer` (for `MarkdownDoc` + node tooltips).
- `ModelPicker` / Settings plumbing.

## 9. Open design questions

1. **Graph density at scale** — dagre top-down is fine for tens of nodes; a 200-lemma
   project may need collapse/cluster (group by file/subnamespace). Revisit when a project
   gets big. *(defer)*
2. **Blueprint authoring ergonomics** — editing raw markdown to add a node is fine for v2.1;
   a structured "add node" form (key/kind/lean/uses fields) could come later. Markdown-first
   for now. *(defer)*
3. **Live graph transport (D42)** — wire the asset signal through the active session's SSE
   into the project window, vs a poll while a project run is active. Start with whichever is
   simpler to land; the poll is the guaranteed fallback. *(decide while building T2/F7)*
4. **In-project session in the sidebar?** — IA says project sessions are hidden from the
   sidebar (only in the window + search). Confirm this still feels right once it's clickable;
   a "recent project sessions" peek is a possible later addition. *(holding the v2 decision)*

## 10. Build plan — vertical slices (interleaved with the backend)

> One slice at a time · backend todo(s) **then** the frontend that consumes them, each slice
> ending in something **clickable on the UI** (the v2 rhythm: plan → your input → build +
> tests → tested → done). Backend todos keep their IDs (P1…V1) from
> `v2.1-projects-architecture.md` §10; frontend todos are **F1…F9**. The single tracker
> `v2.1-progress.html` interleaves them.

- **Slice 1 — Project shell** (create · list · open)
  - `P1` schema · `P2` service + gitstore · `P3` CRUD routes
  - `F1` projects store + Sidebar **Projects group** (D35/D36)
  - `F2` create-project flow + **project window shell** (hero, breadcrumb, tab strip) (D37)
- **Slice 2 — Sessions in a project**
  - `Q1` delete `project.py` + prompt namespace param · `Q2` create-session-in-project +
    bridge context compose · `Q3` chaining smoke
  - `F3` Overview tab: in-project **composer + sessions list**; open a project session →
    Chat+Canvas (D37)
- **Slice 3 — Instructions & Memory**
  - `R1` instructions.md GET/PUT + injection · `R2` memory.md GET/PUT + injection
  - `F4` `MarkdownDoc` component + **Instructions & Memory rail cards** (live memory) (D39)
- **Slice 4 — Files**
  - `S1` upload + store + commit + download · `S2` tiered text extraction
  - `F5` **Files card**: upload dropzone + list + download (D40)
- **Slice 5 — Blueprint & graph** *(the centerpiece)*
  - `T1` blueprint md format + parser + GET/PUT · `T2` derived status + session attribution
    + `/graph`
  - `F6` Blueprint tab **markdown authoring** sub-view (reuse `MarkdownDoc`) (D39)
  - `F7` **dependency graph** (React Flow + dagre; shape/color; click→sessions; live
    refresh) (D38/D42)
- **Slice 6 — Filesystem & sharing**
  - `U1` tree + read + edit-any-file · `U2` export · `U3` `github_token` + remote + push
  - `F8` **Filesystem tab** (tree + viewer/editor) + export + GitHub push UI + **token field
    in Settings** (D40/D43)
- **Slice 7 — Search**
  - `V1` global search endpoint
  - `F9` **⌘K global search overlay** (D41)
