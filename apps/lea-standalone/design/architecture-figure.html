Lea Standalone — Architecture Figure

Current snapshot · apps/lea-standalone (React UI + FastAPI adapter + vendored prover) · 2026-06-29

What this shows. The standalone app as it runs today — the live wiring between the browser UI, the one backend, and the in-process prover, with every input and output labelled at the boundary it crosses. It is a where-are-we-now map, not a roadmap.

The one idea to hold. There is no HTTP boundary between the adapter and the prover — the adapter imports the vendored prover and calls lea.interface.run_events(...) in-process, pattern-matching its typed events onto the browser's SSE stream. One process, one backend port (:8001).

The persistence split is deliberate. Git owns proof content (every file change is committed); SQLite is a rebuildable index (sessions/runs/steps/usage — never proof bytes). A session has no stored status — it is derived from the latest code-step's check verdict, so it can't drift.
