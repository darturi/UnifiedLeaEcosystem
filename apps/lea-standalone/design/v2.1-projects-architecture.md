## 0. Current architecture at a glance

A **project** is a first-class shared workspace — a directory + git repo + DB index that
owns Instructions, Memory, Files, a **blueprint** (the lemma breakdown + dependency graph),
and a set of sessions. The design extends v2's invariants (filesystem-canonical D3, git
owns content D7/D8, DB is an index D4, stateless adapter-authoritative `run` D16, typed
events D17) to a shared, multi-session workspace. (Decision refs in parentheses.)

- **"Project" is an adapter-only concept; the prover stays project-agnostic** (D32). The
  prover already takes `messages` (D16, messages-in) and a `working_dir`; that is all a
  project needs. Project context (goal, instructions, memory, blueprint, files) is just a
  **composed message** the adapter prepends; the working dir is the project repo. The
  legacy prover-side `project.py` (which makes the *prover* regex-parse and write a markdown
  index — prover-side state, violating D16) is **deleted**, not reused.
- **A project = one shared directory + one git repo + one DB index row** (D21). The
  directory **is** the project; `rm -rf` deletes it; its `git log` is the project's whole
  history (one repo for proofs + instructions + memory + files → project-scoped
  git-as-memory, resolving v2 D7 drawback #2).
- **The layout is dictated by Lake, not by legacy** (D22). The lakefile declares
  `lean_lib Lea { srcDir := "proofs" }`, so a module `Lea.<Project>.Foo` *must* live at
  `proofs/Lea/<Project>/Foo.lean` to be importable. Lemmas-that-chain (the user's
  "function-chaining") is therefore a hard requirement that **forces** project proofs to sit
  at `proofs/Lea/<Project>/` (directly under the srcDir — unlike loose sessions at
  `proofs/<uuid>/…`, which are isolated and non-importable on purpose). Non-Lean project
  assets live in a Lake-invisible `.lea/` subfolder in the same per-project repo.
- **Project sessions write into the project repo from turn 1** (D23) — they share the dir
  and namespace, so a new proof can `import`/`open` an already-proved sibling immediately.
  Loose sessions are unchanged. A session belongs to at most one project.
- **The git store resolves the repo from the session's project** (D24): project session →
  the project repo; loose → its own `proofs/<uuid>/` repo. Every git primitive
  (commit-every-write, `git show` snapshot, `diff`, `uncommitted_diff`) is reused unchanged.
- **Instructions, Memory, and the Blueprint are three uniform `.lea/*.md` files** (D25, D26, D28)
  — canonical on disk, versioned in git, read by the adapter and composed (with a file inventory)
  into **one context message** prepended to the run's `messages`. Instructions is user-authored;
  **Memory + Blueprint are co-authored by the agent (via `write_file`/`edit_file`) and the human**.
  No DB columns for their content, no implicit `lea.md`, no prover project code.
- **Files/uploads** (D27) live in `.lea/files/`, are indexed in `project_files`, and are
  text-extracted by tier (native text → pdf/docx → images deferred). The agent reads them
  via the **existing `read_file` tool** (they're in its working dir) + an inventory line in
  context — no RAG in v2.1.
- **The blueprint is a markdown file (canonical); the graph is derived from it** (D28, D29).
  `.lea/blueprint.md` holds the **planned** decomposition — one section per node (definition /
  lemma / theorem), each declaring `kind`, the Lean decl name, and its `uses` dependencies,
  with a prose statement. It is co-authored by the **agent** (via its existing
  `read_file`/`write_file`/`edit_file` tools — no new surface) and the **human**, and
  versioned in git like the canvas (D3/D8). The adapter **parses** it into nodes + edges;
  node **status is derived from the live Lean state** (decl resolves? `check`/`verify` pass?
  `sorry`?) → never stored, can't lie. The render graph (modeled on Lean's `leanblueprint`
  dep-graph: **shape = kind** box/ellipse, **color = status** green/blue/white) is the join of
  the parsed plan + derived status. The blueprint's **proved nodes** are the "which lemmas exist
  and are reusable" knowledge injected into the agent — kept separate from Memory (D26), which is
  just the user's curated notes. No `project_nodes` table; the DB stays a rebuildable index (D4).
- **Project artifacts live in the project window; the session canvas stays Lean-only** (D33).
  The blueprint (graph + markdown view), Instructions, Memory, and Files are project-scoped and
  rendered in the project window — not the per-session Lean canvas. During a run, the agent's
  `.lea/*.md`/asset writes are committed + refresh the graph, but never become canvas snapshots.
- **A Filesystem tab exposes the project git repo** (D34): view + **edit any file** (write +
  commit — the project-wide generalization of v2's writeable canvas), **download** (zip / git
  bundle), and opt-in **commit & push to a GitHub remote** via a redacted `github_token` in
  Settings + a per-project remote URL. Since a project already *is* a git repo, this is mostly
  exposure, not new storage.

**Build status:** design only — nothing built. v2 core (adapter 91 · prover 77) is the
foundation. Build plan in §10; live status in `v2.1-progress.html` (backend + frontend
interleaved). The UI design is in `v2.1-ui-design.md` (D35+).

## 1. What a project is

### D21 — A project is a shared directory + git repo + DB index row (not a filter)
A project owns one on-disk directory that is **its own git repo**, plus a `projects` row
indexing it. Its sessions, proofs, instructions, memory, files, and blueprint all live under
that directory / reference that row. **It is an adapter-side concept end to end** (D32).
- **Why:** the user's framing is "all of this in a single shared project directory." One
  dir = the project: `git log` there is the full history, `rm -rf` deletes it, and a shared
  dir is what lets proofs reference each other (D22). Mirrors v2's "the dir *is* the session"
  (D7), scaled to "the dir *is* the project."
- **Rejected:** *project = a DB tag/filter over loose sessions.* Gives a sidebar grouping but
  no shared directory, shared namespace, cross-proof reuse, project memory, or blueprint —
  i.e. none of the feature. The shared workspace is the whole value.
- **Consequence:** `sessions.project_id` (`NULL` = loose) selects loose vs in-project behavior.

## 2. Filesystem & git layout

### D22 — The layout is dictated by Lake's module resolution
The workspace lakefile declares `lean_lib Lea where srcDir := "proofs"`. Lean resolves a
module `Lea.<Project>.Foo` to the file `proofs/Lea/<Project>/Foo.lean`. **For project lemmas
to chain (import each other) they must therefore live directly under the srcDir as
`proofs/Lea/<Project>/`** — there is no other Lake-valid location. Loose sessions sit at
`proofs/<uuid>/Lea/Misc/…`, whose `<uuid>` segment makes them deliberately *non-importable*
(isolated standalone files); projects are the opposite by requirement.

So a project directory + git repo is:
```
proofs/Lea/<Project>/
  Foo.lean, Bar.lean, …      ← the project's proofs (importable as Lea.<Project>.* → chaining)
  .lea/                      ← Lake-invisible project assets (non-.lean; Lake globs only .lean)
    instructions.md          ← CANONICAL project Instructions, user-authored (D25)
    memory.md                ← CANONICAL Memory, co-authored by agent + human (D26)
    blueprint.md             ← CANONICAL blueprint, co-authored by agent + human (D28)
    files/                   ← uploaded docs + extracted text (D27)
  .git/                      ← the project repo
```
- **Why `proofs/Lea/<Project>/`:** it is the *only* path that satisfies the chaining
  requirement under the given lakefile — a first-principles constraint, derived here, not
  inherited. `<Project>` is a valid Lean module-name segment (UpperCamel, no dashes); the
  adapter derives it from the project slug.
- **Why `.lea/` inside the same repo:** keeps "single shared directory" literal and puts
  instructions/memory/blueprint/files under the **same `git log`** as the proofs. The
  dot-prefix + non-`.lean` contents guarantee Lake never treats it as a module.
- **Rejected:** assets in a sibling `workspace/projects/<slug>/` tree (splits the project
  across two roots + two repos); a per-project Lake *library* with its own srcDir (needs
  lakefile edits per project — pointless complexity when one `Lea` lib already namespaces
  cleanly).
- **Drawback (logged):** the slug → `<Project>` namespace → dir path chain means a slug
  rename is a git-mv + import rewrite. **Slugs are immutable** in v2.1 (rename = title only).

### D23 — Project sessions write into the project repo from turn 1 (no prove-then-promote)
Creating a session inside a project sets its working dir to the project repo; the agent
writes under `Lea/<Project>/…` in the project namespace from the first turn, able to import
sibling lemmas live. Loose sessions are unchanged (own `proofs/<uuid>/` repo, `Lea.Misc`).
- **Why:** matches "sessions start inside the project and share the directory." Proofs
  reference each other immediately; the module name is real from the start; no rewrite step.
- **Rejected — prove-loose-then-promote:** prove in `Lea.Misc`, then move the file into the
  project and rewrite its namespace. The move is **cross-repo** (loose repo → project repo),
  so git history doesn't follow; the module name is throwaway until promotion; and it
  contradicts shared-from-the-start. (This was the legacy `_deferred` workflow; we don't
  revive it. A "move a loose session into a project" convenience may exist later, but built
  clean, and never as the primary path.)
- **Concurrency:** many sessions share one repo, but v2's global one-run lock
  (`active_run_lock`) serializes all writes app-wide, so per-project contention is moot today
  (Open Q-5).

### D24 — The git store resolves the repo from the session's project
v2's `GitStore` keys operations on a repo path derived from `session_id`. Generalize the
resolution: **project session → the project repo (`proofs/Lea/<Project>/`); loose → its own
repo.** A single `repo_for_session(session) -> Path` resolver (fed `project_id` + the
project's `<Project>`) is called by the bridge and the file/check/verify routes; every
primitive (`commit_write`, `snapshot`, `diff`, `head`, `uncommitted_diff`) operates on the
resolved path, unchanged.
- **Per-session vs project history:** the per-session view is `code_steps` filtered by
  `session_id` (the DB is the per-session index); the project repo's `git log` is the
  project-wide, interleaved view (intended).
- **Why:** the repo boundary is the *only* thing differing between loose and project work;
  isolating it in one resolver keeps the bridge and routes project-agnostic.

## 3. Context the agent receives — Instructions & Memory

### D25 — Project context is one composed message the adapter prepends (not prover code, not `lea.md`)
`run_events` is messages-in (D16/A9). So project context — the goal/Instructions, Memory,
the blueprint (what's proved / what's planned / how it chains), and the file inventory — is
assembled by the adapter into **one context message** prepended to the run's `messages`.
**Instructions, Memory, and the Blueprint are three uniform `.lea/*.md` files** — canonical on
disk, versioned in git, read by the adapter and folded into the context. Instructions
(`.lea/instructions.md`) is the user's project goal + rules: **user-authored** (the agent reads
it but doesn't rewrite it), edited in the project window. (Memory and Blueprint are *also*
agent-written — D26/D28.)
- **Why:** it's the cleanest possible seam — no new prover surface, no implicit file
  discovery, deterministic and per-run. It honors D16 (adapter composes the input; prover is
  pure compute) and D32 (prover knows nothing about "projects").
- **Rejected:** the implicit `lea.md` discovery in `prompt.py` (order-dependent, global, not
  per-project); the legacy `project.py` `project_context_message` + markdown-marker parsing
  (prover-side, brittle regex, violates D16). Both are replaced by "compose a message."
- **Namespace/location guidance:** the composed message states the write location
  (`proofs/Lea/<Project>/`) and namespace (`Lea.<Project>`) explicitly. The system prompt's
  hardcoded `Lea.Misc` workspace block is **parameterized** (a small, clean prover-prompt
  change — D32) so it states the active namespace instead of always `Lea.Misc`.

### D26 — Memory = `.lea/memory.md`, co-authored by the agent + the human (a markdown file)
Memory is the markdown file **`.lea/memory.md`** — a living record of useful facts, learnings,
and user preferences (e.g. "prefer explicit witnesses for ε given δ", "the √x attempt failed on
the unbounded domain — try splitting at x=1"). **Both the agent and the human write it:** the
agent edits it during a run with its existing **`write_file`/`edit_file`** tools (it's told to
record durable learnings — what worked, what failed, preferences it was given), and the human
edits it in the project window. Committed on every write (D8), versioned in git, injected into
the run context (D25). Plain markdown — a bulleted list is the natural shape; "nothing fancy."
- **Why a file, not a DB table:** same first-principles seam as the Blueprint (D28) and
  Instructions (D25) — the agent's native interface is files, so a `.md` it can read+write needs
  *no new tool or prover surface*; the human edits the same file; git versions it; the DB stays a
  pure index (D4). Memory, Instructions, and Blueprint are three uniform `.lea/*.md` docs.
- **No `kind`/`source` schema, no `project_memory` table** — the file is canonical. (This is the
  resolution of the old "list of strings" idea: it's just the markdown content, and making it a
  file is what lets the agent maintain it too.)
- **Decoupled from "what's proved":** the reusable-lemma facts (which lemmas exist + are proved,
  how to import them) are the blueprint's proved nodes (D28), derived from live Lean. Memory is
  for the softer, human/agent-curated notes that aren't mechanically derivable.

## 4. Files & uploads

### D27 — Files in `.lea/files/`, indexed in DB, tiered text extraction, read via `read_file`
Uploaded docs (`.pdf/.tex/.docx/images`) are saved to `.lea/files/` (versioned in the project
repo) and indexed in `project_files` — `id, project_id, filename, stored_path, mime,
kind ('upload'|'blueprint'|'extract'), extracted_path, created_at`. Extraction is **tiered**:
- **Tier 1 — native text** (`.tex/.md/.txt`): used directly.
- **Tier 2 — extractable** (`.pdf/.docx`): extracted to `.lea/files/<name>.txt` at upload
  (pdfminer.six / python-docx).
- **Tier 3 — images** (`.png/.jpg`): stored + downloadable, **not** extracted (vision/OCR is a
  later, model-side concern).

The agent reads extracted text through the **existing `read_file` tool** (the files are in its
working dir); the composed context (D25) carries a one-line **inventory**. No embeddings/RAG.
- **Why phased + read-file, not RAG:** simplest thing that works. Native text is free; pdf/docx
  are the common math sources; vision deferred. On-demand `read_file` respects the context
  budget and reuses an existing tool — no retrieval path until file volume forces one (Open Q-2).

**Upload mechanism — the "how" (this is the one net-new I/O path in v2.1):**
1. **Browser:** a file input / drag-drop in the Filesystem tab + the rail Files card POSTs
   `multipart/form-data` to `POST /api/projects/{id}/files`. (Plain HTML multipart — no special
   client lib; the existing `lib/api.ts` fetcher sends a `FormData` body.)
2. **Adapter:** FastAPI receives it as `UploadFile = File(...)`, which **streams to a spooled
   temp file** (large PDFs never sit fully in memory). This needs the **`python-multipart`**
   dependency added to the adapter (plus `pdfminer.six` + `python-docx` for Tier 2).
3. **Pipeline (one flow):** validate MIME against an allowlist + enforce a size cap → sanitize the
   filename (no path escape; de-dupe collisions) → write bytes to `.lea/files/<name>` →
   `commit_write(author="user")` (D8 — git stores binaries fine) → extract text by tier into a
   `.lea/files/<name>.txt` sidecar → insert the `project_files` row(s) → return the file metadata.
4. **Download** is the inverse: `GET …/files/{fid}` streams the stored bytes (`FileResponse`).
5. **Surfacing:** the file shows up in the Filesystem tree (D34) + the rail Files card; its
   extracted `.txt` is what the context inventory points the agent at.
- **Guards:** size cap + MIME allowlist (no executables/scripts) + filename sanitization + the
  same path-escape guard used by the file-write endpoint (D34).
- **Binaries in git (tradeoff, logged):** committing PDFs/images versions them and makes them part
  of the GitHub push + download — matching the "shareable filesystem" intent and the convention in
  math/`leanblueprint` repos. It does bloat git history; Git-LFS or an external blob store is a
  future option if a project accumulates large media (Open Q-2).

## 5. The blueprint & dependency graph (the visualization)

### D28 — The blueprint is a canonical markdown file; the graph is parsed + derived from it
The blueprint is **`.lea/blueprint.md`**, the single canonical source for the project's
decomposition. It is a real document (the "tex/overleaf" artifact of the search→tex→Lean
loop), co-authored by the **agent** (via its existing `read_file`/`write_file`/`edit_file`
tools — *no new tool or prover surface*) and the **human**, committed to the project repo on
every write (D8) so its evolution is git history. Format — one `## ` section per node, a small
key:value header, prose statement:
```markdown
# Blueprint — Epsilon

## continuous_sq
- kind: lemma                         → node SHAPE (definition=box · lemma/theorem=ellipse)
- lean: `Lea.Epsilon.continuous_sq`   → links the node to its live Lean decl (omit until named)
- uses: tendsto_iff_eps, distance_lower   → dependency edges (the chaining; declared)

The function x ↦ x² is continuous in the ε–δ sense.     ← statement prose (NL/LaTeX)
```
- **Parsing is a robust per-section line-scan** (split on `## `; read `- key: value` header
  lines; the rest is the statement) — *no regex over free prose*. The adapter parses it into
  nodes + `uses` edges on read.
- **Status is derived, never written in the file** → node **color**: `planned` (no `lean:`
  yet / decl absent, white) · `stated` (decl exists, proof has `sorry`, blue-ish) · `ready`
  (all deps proved — derived from edges, blue) · `proved` (`check`/`verify` ok, no `sorry`,
  green) · `failed` (last attempt errored). Derived from the live Lean state, so it can't lie.
- **Why markdown-canonical (not DB-canonical):** the agent and human edit the blueprint with
  the tools they already use on files; it's git-versioned in the shared dir; and the DB stays a
  rebuildable index (D4) rather than a second source of truth. This matches v2's
  filesystem-canonical / DB-as-index stance (D3/D4) and the ecosystem convention (leanblueprint
  / numina / MCP-skill setups all use a blueprint *file*).
- **Why a blueprint at all (not "graph from finished proofs"):** the value is showing *what
  isn't done yet* and how the pieces connect — "understand what the agent is trying to
  accomplish." It is also the agent's task decomposition: it's in the injected context (D25) so
  the agent knows which proved lemmas to reuse and which node it's attacking.
- **Exposed as** `GET /api/projects/{id}/graph -> {nodes(kind,status), edges}` (parse + derive)
  and `GET …/blueprint` (raw markdown for the authoring view). Frontend renders shapes/colors.
- **Rejected:** DB-canonical `project_nodes` (needs a new mutation path the agent can't reach
  with file tools; a second source of truth to keep in sync); the legacy *prover-side* markdown
  index (violates D16/D32); full leanblueprint **LaTeX** parsing (heavy — markdown is enough,
  and a `.tex` *import* path can come later, Open Q-3).

### D29 — Authoring (agent + human), derived status, parser-only adapter
- **Authoring:** the agent maintains `.lea/blueprint.md` as part of its work — the composed
  context (D25) tells it the file's location + format and to add a node when it plans a lemma
  and set/confirm the `lean:` name when it formalizes one. The human edits the same file (the
  authoring view writes it back). Both are normal file writes (committed, D8).
- **Status derivation:** computed on read — resolve each node's `lean:` decl in the project
  files, run the existing `check`/`verify` capabilities (v2 D2) + a `sorry`/`axiom` scan, and
  fold in dependency readiness from the edges. Cached only as a rebuildable convenience, never
  authoritative.
- **Session attribution is derived (node ↔ sessions).** Each node carries which sessions worked
  on it, so the graph is navigable back into the chat. It's **derived from `code_steps`**, not
  stored: locate the node's `lean:` decl in the project files (scan for `theorem/lemma/def <name>`
  → its file), then every `code_step` that committed that file (each has `session_id` + `commit_sha`,
  v2 C1/C4) is a session that touched the node — newest = `last_modified_by`. So "session 123
  formalized lemma x" is reconstructed from git + the DB index, with no node↔session table to keep
  in sync. The `/graph` node payload includes `sessions: [...]` + `last_modified_by`; clicking a
  node opens those sessions. (Resolves Open Q-4's "map sessions to the node they changed.")
- **The adapter never *writes* the blueprint** — it parses on read and derives status. This
  keeps a single writer model (agent + human, like the canvas), honors D16/D32 (no
  prover-written index; no adapter-fabricated plan), and means there's no `record_project_entry`
  step: the git-committed `.lean` files + `.lea/blueprint.md` are the whole source of truth.
- **Edge case (logged):** if the agent proves something it forgot to put in the blueprint, the
  proof still exists (as a session + git commit) but won't appear as a graph node until the
  blueprint is updated. The injected blueprint gives the agent a standing incentive to keep it
  current; a "proved-but-unlisted" reconciler is a possible later nicety (Open Q-3).

### D33 — Project artifacts live in the project window; the session canvas stays Lean-only
The session **canvas is session-scoped and Lean-only** (it shows the proof the current session
edits). The blueprint, Instructions, Memory, and Files are **project-level** artifacts surfaced
in the **project window** (the mockup's graph + right-rail cards), *not* the session canvas — so
the canvas never needs to become a generic multi-file editor (v2-ui Open Q-6 stays unneeded).
- The blueprint has two project-window views: the **graph** (derived, D28) and a **markdown
  authoring view** (raw `.lea/blueprint.md` via GET/PUT). Instructions/Memory/Files are the rail
  cards. None of these touch the per-session canvas.
- **During a run**, the agent's writes to `.lea/blueprint.md` (and other assets) are *not* canvas
  code_steps. v2 already emits the meaning-level `FileChanged` (→ canvas snapshot) **only for
  `.lean`** (A2), so asset writes never become snapshots automatically. The bridge detects an
  asset write from the generic `ToolCalled(write_file/edit_file, path=…non-.lean)`, **commits it**
  (the git layer's `git add -A` covers any file) and emits a light **"blueprint updated"** signal
  that refreshes the project graph — no new prover event, no canvas pollution.
- **Why:** session and project are different scopes; the blueprint is the project's spine, not one
  session's artifact. This is the direct answer to "the canvas is Lean-only" — we don't fight that;
  we put project `.md` artifacts where they belong (the project window) and keep the canvas simple.

## 6. Database — schema deltas

### D30 — Schema deltas from the v2 `db.py` (clean rebuild, no migrations)
Continues v2's disposable-DB / create-table-authoritative stance (no `ALTER` migrations; a
schema change = fresh DB via `npm run reset:local`).

**`projects`** — extend:
```
id text pk · slug text unique (immutable; → namespace Lea.<Project>) · title · description
namespace text not null           -- cached 'Lea.<Project>' (derivable; cached for queries)
repo_path text not null           -- 'proofs/Lea/<Project>' (the shared dir / repo, D22)
remote_url text                   -- per-project GitHub remote for push (D34; nullable)
created_at · updated_at
```

**No `project_nodes`, no `project_memory`, no `instructions` column** — Instructions, Memory, and
the Blueprint are the three canonical **`.lea/*.md` files** (D25/D26/D28), parsed/read on demand;
the DB never stores their content (filesystem-canonical D3, DB-as-index D4). `description` stays a
column only because it's short metadata shown in the project list/cards.

**`project_files`** (new) — D27 uploads:
```
id · project_id (fk) · filename · stored_path · mime
kind ('upload'|'blueprint'|'extract') · extracted_path · created_at
```

**`sessions`** — `project_id` FK exists. Listing splits **loose** (`project_id IS NULL`, the
sidebar Chats group) vs **in-project** (the project window).

## 7. API surface (adapter)

### D31 — New/changed endpoints
```
GET    /api/projects                          list (title, slug, namespace, proof/node counts, status mix)
POST   /api/projects                          create {title, description?, instructions?} → slug, dir, repo
GET    /api/projects/{id}                      detail: meta + sessions + memory + files + blueprint
PUT    /api/projects/{id}                      edit title/description (metadata only)
DELETE /api/projects/{id}                      rm -rf the repo + cascade DB rows (confirm guard)

POST   /api/projects/{id}/sessions             create a session INSIDE the project (working dir = project repo, D23)
```

## 8. Prover-side: stays project-agnostic

### D32 — Delete the prover's project machinery; the prover knows nothing about "projects"
A project needs only two things the prover **already** provides: messages-in (D16/A9) and a
`working_dir` (threaded by the bridge in v2). So:
- **Delete `prover/lea/project.py`** (`ProjectContext`, `project_context_message`,
  `record_project_entry`, the markdown-marker parsing) and the **`project=` parameter** of
  `run_events`. They make the prover hold/produce project state (a prover-written markdown
  index, parsed by regex) — a direct violation of v2 D16 (adapter is the single source of truth;
  prover is pure compute) and the reason the legacy seam was brittle.
- **One small, clean prover change:** parameterize the system-prompt workspace/namespace block
  (currently hardcoded to `Lea.Misc`) so the adapter can state the active namespace + write
  location. This is a clean parameter, not a workaround.
- **Everything else is adapter-side:** context composition (D25), result recording (D29),
  blueprint/graph (D28), git (D24). The prover compiles and proves; the adapter owns "project."
- **Why this is cleaner, not just different:** it removes ~390 lines of prover-side
  regex/markdown state, makes the prover's contract uniform (loose and project runs differ only
  in the messages + working dir the adapter hands it), and keeps the D16 boundary honest.

## 8A. Filesystem tab — view, edit, download, share to GitHub

### D34 — A project repo is already a git repo, so "Filesystem + share" is mostly exposure
Because a project **is** a git repo on disk (D21/D22), the Filesystem feature is largely about
*exposing* what already exists — not new storage. The project window gets a **Filesystem tab**
(name TBD) with the full file tree of `proofs/Lea/<Project>/` (proofs, `.lea/` assets, uploads),
each file viewable + **editable** in place, plus **download** (zip / git bundle) and **commit &
push to a user-supplied GitHub remote**. So a mathematician can see everything in one place and
share the whole project with collaborators.
- **Edit any file:** `PUT /api/projects/{id}/file {path, content}` writes under the repo
  (path-escape guarded, as the v2 canvas-edit endpoint D3 already does) and **commits** (D8,
  `author='user'`). Editing a `.lean` re-uses the standalone `check` (v2 D2/D4) for a verdict;
  editing `.lea/blueprint.md` refreshes the graph (D33). This is the *project-wide* generalization
  of v2's writeable canvas — same write+commit primitive, any file, not just the session's `.lean`.
- **Download:** `GET …/export` streams a zip (or `git bundle`, preserving history) of the repo.
- **GitHub push (opt-in, user-driven):** `PUT …/git/remote` sets the project's remote;
  `POST …/git/push` does `git push` to it using a **GitHub token from Settings** (below). Pushing
  is an outward-facing action → **always explicit** (a button, never automatic), and the UI
  confirms the target repo before the first push.
- **GitHub token in Settings:** add a `github_token` field to `config/lea.local.toml` +
  `/api/settings` (validated by regex like the provider keys), exported to env / held by the git
  layer **only** for push. Like provider keys, it is **never** returned to the client or logged
  (settings reports presence, not value — v2's `configured_provider_keys` pattern). Per-project
  remote URL lives on the `projects` row; the token is global in settings.
- **Why this fits cleanly:** no new content store (the repo exists), no new write primitive
  (commit-on-write exists), and git push is the natural "share" since the project is already
  versioned. The only genuinely new surfaces are a directory **listing/read/export** and the
  **push** call + token plumbing.
- **Guards:** writes path-escape-checked + committed; `.git/` is read-only/hidden in the tree;
  push is rate-limited to explicit user action; the token has its own validator + redaction.
- **Open:** whether "Filesystem" also lists the *Lean build* artifacts (it should hide
  `.lake/`, `.git/`, oleans — show source + assets only). (Open Q-7.)

## 11. Open questions — **resolved** (decisions logged)

All seven are decided; the design above reflects them. Kept here as the decision record.

1. **Agent-written memory (D26):** ✅ **Both agent + human write Memory.** Memory is a markdown
   file `.lea/memory.md`; the agent edits it with `write_file`/`edit_file` (no special tool). This
   folded Memory into the uniform `.lea/*.md` model (dropped the `project_memory` table).
2. **File context + binary storage (D27):** ✅ **read-file-on-demand + inventory (no RAG); commit
   binaries straight into the project git repo.** Git-LFS only if media bloats history later.
3. **Blueprint extras (D28/D29):** ✅ **Defer both** the leanblueprint `.tex` import and the
   "proved-but-unlisted" auto-reconciler. Markdown authoring + the agent keeping it current suffices.
4. **Graph granularity + session attribution (D28/D29):** ✅ **Lemma/theorem-level nodes**, plus a
   **derived node ↔ sessions mapping** (which session formalized/modified each node — from
   `code_steps`, no stored table), so the graph links back into the chat (D29).
5. **Concurrency in a shared repo (D23/D24):** ✅ **Out of scope.** The global one-run lock makes it
   moot; deliberately not handling concurrent runs in v2.1.
6. **`project.py` deletion timing (D32):** ✅ **Delete in Q1**, up front, so the adapter-only
   boundary is honest from the start.
7. **Filesystem tab scope (D34):** ✅ **Source + assets only** (hide `.git/`, `.lake/`, oleans);
   **whole-project → one remote** for v2.1 (token global in Settings, remote URL per-project).
