Lea UI - System Architecture

1 · Web UI — the React front end (Vite dev server on :5173)

Lean canvas

The live view of the current .lean proof file. The agent writes here and the user can co-edit the same file, then trigger a lean_check (fast compile) or a full verify (SafeVerify). Backed by the session file + check + verify endpoints (/sessions/{id}/file · lean-check · verify).

Project window

The project-context surface that wraps a proof in its supporting material: the blueprint graph, free-text instructions, persistent memory, and the project's files. Served by the project routes (/api/projects/*).

Settings

Configuration: choosing the provider / model, managing API keys (stored redacted, presence-only to the client), and viewing token usage / cost. Backed by /api/settings, /api/models, and /api/stats.

2 · The backend — one OS process, no internal HTTP

Adapter API

The single backend process and the only HTTP surface in the system. It exposes everything the browser needs under /api/* — runs, sessions, projects, settings, models, stats, plus the per-session file / lean-check / verify endpoints — and turns each run into a long-lived Server-Sent Events response. It owns the supervision policy and imports the prover as a library, so there is no separate "Lea API" service; the prover runs inside this same process (FastAPI / uvicorn).

Bridge

The in-process seam between the adapter and the vendored prover — there is no socket between them. For each run it composes the session/project context into the prover's messages, builds its LeaConfig, and calls run_events(...) directly as a Python generator. It then pattern-matches the prover's typed events (TurnStarted, ToolCalled, FileChanged, CheckResult, VerifyResult, UsageUpdated, ToolApprovalRequested, Finished, …) onto the SSE stream, applies the approval gate, and persists results — a git commit on every FileChanged plus an index row in SQLite.

Approval gate

A per-tool checkpoint the Bridge applies to the prover's tool calls. Read-only / no-effect tools (read_file, lean_check, search_mathlib) are auto-approved and never pause. Side-effecting tools (write_file, edit_file, bash) are gated: the run parks, a ToolApprovalRequested prompt rides the SSE to the UI, and the prover only proceeds once the user allows or denies (with an "always allow this session" option). Autonomous runs (the Overleaf path) disable the gate entirely.

3 · Lea — the Lean 4 theorem-proving agent

Agent loop

The proving loop at the heart of Lea. It runs a single tool-calling cycle — write a .lean file, compile it, read the errors, edit, repeat — streaming model tokens from the providers until the proof compiles with zero errors and zero sorrys. Exactly six tools, no hidden orchestration.

The six tools

The agent's entire surface. Read-only — read_file, lean_check, search_mathlib — run freely; write / side-effecting — write_file, edit_file, bash — pass through the approval gate first. lean_check compiles via the LSP daemon, and search_mathlib looks up Mathlib lemmas under a fixed per-problem budget.

LSP daemon

A persistent lake env lean --server per Lake root that keeps Mathlib's compiled oleans warm. It's what makes lean_check fast — roughly 0.2s for an in-place edit versus ~88s cold — so the write → check → edit loop stays interactive.

SafeVerify

The honest "verified" verdict. After a proof compiles, SafeVerify replays it through the Lean kernel and checks each declaration's type/body against an axiom whitelist, catching structural cheats — namespace shadowing, sorry-in-imports, redefining a name as True — that a plain compile + sorry-grep miss. It's read-only and runs separately from the agent's tool loop.

Lean + Mathlib

The Lake workspace (Lean and Mathlib both pinned via lean-toolchain) where proofs are actually written and compiled, under proofs/. It's the local resource every tool ultimately touches: lean_check and SafeVerify replay against it, search_mathlib queries it, and write_file / edit_file mutate its files.

4 · Data stores — a deliberate split: content vs. metadata

Git store

Git owns the proof content. Each session/project is a git repo, and the Bridge commits on every FileChanged; a timeline row keeps only the resulting commit_sha + path, not the file bytes. The Lean canvas reads history back through the adapter via git show, and user canvas edits are committed too (linked to an edit_note message). Keeping the filesystem canonical is also what lets the LSP fast path stay warm.

SQLite index

A rebuildable index of metadata — never proof bytes, and never the canonical .lea/*.md content (instructions / memory / blueprint live as files in the project repo). Because a session's status is derived from its latest code-step verdict rather than stored, it can't drift. It's a clean v2 schema (no in-place migrations — disposable, single-user):

5 · External — actors and resources outside the process

Users

The human collaborator (a mathematician). They supply the natural-language statement, the project context, and the model + keys, then steer the run and answer the approval prompts; in return they get a compiling proof (zero errors / zero sorrys), live progress, the check + SafeVerify verdicts, and a downloadable repo.

Model providers

Gemini · Anthropic · OpenAI, reached over HTTPS — the only outbound internet dependency. The agent loop streams tokens from whichever provider/model is selected; keys come from the root .env / Settings and are kept redacted (presence-only to the client).

Model catalog

Two layers. A small curated shortlist (models.json in packages/lea-model-catalog) is the default pick list shared by both front ends and the adapter, so it stays consistent across the standalone UI and the Overleaf path. The searchable picker in Settings is LiteLLM-backed (models_catalog.py): the full LiteLLM catalog — roughly 2k chat models across ~76 providers — which also drives the dynamic API-key prompt (which env var a chosen model needs). LiteLLM is optional and degrades gracefully: if it can't be imported, the adapter falls back to the curated shortlist. Served to the browser via /api/models.

Next directions — where this is headed

1 · No axiom and opaque

Axioms (and opaque declarations) are escape hatches — they let the agent "close" a goal without actually proving it. Axioms are already caught: SafeVerify collects every axiom a declaration depends on and rejects anything outside its whitelist (so a stray axiom or a sorry fails verification). The gap is opaque — an agent-introduced helper opaque isn't an axiom, so it slips past that check unless it happens to clash with a target declaration's kind. The next step is to extend the same hard stop to cover agent-introduced opaque (alongside the existing sorry / native_decide bans), enforced at verify time rather than trusted to the prompt.

2 · Human-in-the-loop formalization — a workflow proposed by Prof. Jaume

Drop in a file, and the first step is to break it down into its definitions, lemmas, and theorems. For each one, run a deep search against Mathlib to decide what already exists, and report two lists: what is already formalized (reuse it) and what must be formalized from scratch.

From there, formalize the pieces one at a time with a human in the loop — the definitions, lemmas, and theorems the whole proof rests on get expanded and signed off by a person before anything builds on them. Items that merely compose existing building blocks can be drafted autonomously (at least the first pass). Once the foundations are settled, the final proof can be handed off to the agent — and even that initial draft can run autonomously, provided the agent first relays back every assumption it made, so nothing is silently baked in.

3 · Testing on definition-heavy samples

For evaluation, lean toward definition-heavy samples rather than proof-heavy ones — they exercise the decompose / Mathlib-search / formalize-from-scratch path that matters here, instead of just stress-testing tactic search. We need a small suite of 4–5 examples to test against, and to compare different prompts and configurations across them.

4 · Human-readable refinement — few-shot from Prof. Jaume's style

Prof. Jaume generally cleans up the Lean to make it more human-understandable, and when explaining me a proof he wrote it in a natural-language, algorithm-style form. We can capture that style — together with a few samples from his repo — as few-shot examples in the prompt, applied as a dedicated "refining" step that runs after the initial draft compiles: the goal there is a final proof that is readable, not just correct.

5 · Side-by-side NL ↔ Lean view — future integration, suggested by Prof. Jaume

A split editor that places the Lean file / theorem on one side and the Overleaf (natural-language) source on the other, so the informal statement and its formalization sit next to each other — making it easy to see which prose maps to which Lean declaration as the proof is built.
