## 0. Current architecture at a glance

A **factory** is a reusable, user-curated catalog of capabilities that can be **attached
to projects** — so house rules and external tools are authored once and shared across the
projects that want them. Two factories, one mental model:

- **Skill factory** — a catalog of **skills**: named markdown fragments (tactic recipes,
  naming conventions, project house rules). The prover injects each into the system prompt
  (`prover/lea/skills.py` → `prompt.py`); it is prose the agent reads, **not** a tool.
- **MCP factory** — a catalog of **MCP servers**: named server specs (stdio `command/args/env`
  or remote `url/headers/transport`). The prover starts them per-run and registers their tools
  (`prover/lea/mcp.py`); they become **callable tools** the agent can invoke.

The design extends v2.1's invariants (adapter-only concepts D32/D44, DB-is-an-index D4,
typed events D17, per-run prover contract D16) to two shared, cross-project catalogs.
(Decision refs in parentheses.)

- **Factories are adapter-side catalogs over the prover's existing support** (D44). The
  prover stays unchanged — it already takes `cfg.skills` (paths) and `cfg.mcp_servers`
  (specs). The adapter owns the catalog, the scoping, and the run-time *resolution* that
  fills those two fields. The opening is the existing **wiring gap**: today
  `adapter/app/config.py` builds `LeaConfig` but leaves both empty.
- **A skill is a DB row, not a git file** (D45). `skills(id, name, slug, body, is_global, …)`
  holds the markdown **body in a column**; non-global assignment lives in a
  `skill_projects` join. This is a **deliberate departure** from v2's "git owns content"
  (D7/D8): skills are small, user-authored, config-like prose — not proof artifacts the
  agent writes and we diff. At run time the body is **materialized to a temp `.md`** (the
  prover wants file *paths*), outside any git repo.
- **An MCP server is a DB row with redacted secrets** (D46). `mcp_servers(id, name, spec,
  is_global, …)` holds the spec; secret material (`env` tokens, `headers`) is stored and
  surfaced through the **same mask/validate/redact path** the provider keys and the
  `github_token` use (D34/U3 — presence-only to the client, never logged). Non-global
  assignment lives in an `mcp_server_projects` join.
- **One scope model serves both** (D47): `is_global = true` → applies to **all projects**;
  `is_global = false` → applies only to the projects in the join table. **Loose (non-project)
  sessions get neither** — they are deliberately one-off, clean temp work. A future
  per-session manual picker can opt specific factory units into a loose session.
- **Resolution happens at run start, in the bridge** (D48), where `session → project` is
  known. Project session → `global ∪ that project's assigned units`. Skills are written to
  per-run temp `.md` files → `cfg.skills`; MCP specs are merged (secrets decrypted/injected)
  → `cfg.mcp_servers`. Loose session → both stay empty (the no-op of today, now intentional).
- **MCP tools are gated by default** (D49). The interactive approval gate today hardcodes
  `{bash, write_file, edit_file}` (`bridge.py`). Any tool surfaced by an MCP server in a run
  is **added to that run's gated set**, so external tools require human approval. Autonomous
  (Overleaf) runs keep the gate off, exactly as today (D19).
- **The primary way to add a unit is a GitHub link** (D56/D57). Skills and MCP servers are
  shared as repos, so the headline gesture is **paste a GitHub URL → Add → choose scope** (all
  projects / select few). The adapter fetches via a **shallow clone** (reusing the `github_token`
  from Settings, D34/U3, for private repos + rate limits; public repos import tokenless). A
  `source_url`/`source_ref` is stored on both tables so a unit can be **re-synced** from source
  later. **Skills import is one-click** (markdown → `body`); **MCP import prefills the launch
  command** from the repo and the user **confirms** it (D57) — that confirm *is* the trust
  boundary, since the launch command is arbitrary local exec. Manual authoring (no URL) stays
  supported.
- **Schema + API are additive** (D50): two tables + two join tables; CRUD + assignment routes
  under `/api/skills` and `/api/mcp-servers`. The DB stays a rebuildable index (D4) — nothing
  here is derived state that can drift, except the per-run resolution, which is recomputed
  every run.

**Build status:** design only — nothing built. v2.1 Projects (adapter 186 · graph audited)
is the foundation. Build plan in §10; live status in `v2.1.1-progress.html`. The UI design
is in `v2.1.1-factories-ui.md` (D51+).

## 1. What a factory is

### D44 — Factories are adapter-side catalogs over the prover's existing skill/MCP support
The prover already injects `cfg.skills` (markdown paths → system prompt) and starts
`cfg.mcp_servers` (specs → registered tools) per run. The adapter is the **only** loader of
`LeaConfig` (`adapter/app/config.py`) and today populates **neither**. A factory is the
adapter-side catalog + scoping + the resolution that fills those fields per run.
- **Why:** it reuses the agent's real, tested machinery (no new injection path, no new tool
  lifecycle) and keeps the D16/D32 boundary honest — the prover stays capability-agnostic;
  "which skills / which servers apply to this run" is an adapter concern, like project context.
- **Rejected:** *teach the prover about a catalog / scoping.* Duplicates the Projects mistake
  the legacy `project.py` made (prover-side state). The prover should keep taking a flat
  `skills` list + `mcp_servers` dict and nothing more.
- **Consequence:** zero prover changes; all work is in the adapter (`config`/`bridge` +
  new `skills.py`/`mcp_catalog.py` services + routes) and the UI.

## 2. Data model

### D45 — A skill is a DB row (markdown body in a column), not a git file
```
skills(id, name, slug, body, is_global, created_at, updated_at)
skill_projects(skill_id, project_id)        -- only for non-global skills
```
`body` is the raw markdown. `slug` is the stable identifier (and the materialized filename
stem, so the prover's `## Skill: <stem>` header reads cleanly). `is_global` toggles
all-projects vs. join-listed.
- **Why a table, not git (D7/D8):** "git owns content" exists for **proof bytes the agent
  writes and we diff** — history matters there. A skill is small, user-authored, config-like
  prose; a DB row is simpler, queryable, and has no agent-write/diff requirement. This is a
  conscious, scoped exception, logged here so it's intentional, not drift.
- **Materialization:** at run start the adapter writes each applicable skill's `body` to a
  **per-run temp `.md`** (a tempdir, **not** inside any project repo — avoids polluting git),
  and passes those paths as `cfg.skills`. The files are removed when the run ends. The table
  stays the single source of truth; the file is a throwaway the prover reads.
- **Rejected:** *store skill markdown as `.lea/skills/*.md` in each project repo.* Couples a
  reusable, cross-project unit to one repo and forces copy-on-assign; the whole point is
  author-once-share-many. A central table + join is the natural shape for many-to-many.

### D46 — An MCP server is a DB row with redacted secrets
```
mcp_servers(id, name, transport, spec_json, is_global, created_at, updated_at)
mcp_server_projects(mcp_server_id, project_id)   -- only for non-global servers
```
`spec_json` holds the prover-shaped spec — stdio `{command, args, env, cwd}` or remote
`{url, headers, transport}`. **Secret material** (`env` values, `headers` values) is handled
by the **existing mask/validate/redact path** (the provider keys + `github_token`, D34/U3):
stored server-side, **presence-only** in any payload to the client, never logged.
- **Why a table (not git):** an MCP server is **config, not content** — there is nothing to
  diff or co-author. It belongs with settings, alongside the keys it often carries.
- **Why reuse the redaction path:** MCP `env`/`headers` routinely carry API tokens; they must
  get the same never-echo treatment as provider keys, not a new bespoke one.
- **Tool naming:** the prover already registers MCP tools as the bare tool name, falling back
  to `<server>__<tool>` on a clash (`mcp.py`). The adapter doesn't rename; it only needs the
  **resulting tool-name set** for gating (D49).
- **Rejected:** *one `mcp.json` config file shared globally.* No per-project scoping, no
  redaction story, and editing JSON by hand is exactly the friction the factory removes.

## 3. Scoping

### D47 — One scope model: global (all projects) | per-project assignment | loose-gets-none
Both factories share it. `is_global = true` → the unit applies to **every project**.
`is_global = false` → it applies only to the projects listed in its join table. **Loose,
non-project sessions resolve to neither** by default.
- **Why loose = none:** a loose session is intentionally one-off, throwaway work; silently
  attaching global house-rules/servers would change its behavior against the user's mental
  model ("this is just a quick scratch proof"). Keeping loose clean is the point.
- **"Global" means all *projects*, not all *sessions*.** The word is scoped to the project
  axis on purpose (see above).
- **Future (noted, not built):** a **per-session manual picker** that lets a user opt specific
  factory units (already authored in the factory) into a single loose session. Logged in §10A.
- **Rejected:** *a single global flag that also blankets loose sessions.* Violates the loose =
  one-off intent. *Per-project copies instead of a global flag* — forces re-assigning on every
  new project; a flag is the clean "applies everywhere" primitive.
- **Consequence:** skills/MCP only resolve when `sessions.project_id IS NOT NULL`, which keeps
  the near-term wiring small.

## 4. Run-time resolution & wiring

### D48 — Resolve the applicable set in the bridge; populate `cfg.skills` + `cfg.mcp_servers`
The bridge already knows the session, its project, and composes project context
(`bridge.py`). It is the single place to resolve and inject factory units, just before
`run_events(cfg, …)`.
- **Algorithm (per run):**
  1. If the session has no `project_id` → skills `[]`, mcp `{}` (loose, D47). Done.
  2. Else gather `global ∪ assigned` for the project, for each factory.
  3. **Skills:** write each `body` to a per-run temp `.md` (D45) → `cfg.skills = [paths]`.
  4. **MCP:** merge specs, inject decrypted secrets → `cfg.mcp_servers = {name: spec}`.
  5. Compute the **MCP tool-name set** for gating (D49).
  6. Run; in `finally`, delete the temp skill dir (the prover already unwinds MCP servers).
- **Why the bridge, not `config.py`:** `config.py` builds the *base* `LeaConfig` before a
  session/project is known; resolution is **per-run and project-dependent**, so it belongs at
  the run seam where Projects context already lands (D25). Base config stays project-agnostic.
- **Rejected:** *resolve once at config load.* Wrong lifetime — a server can be assigned to a
  project between runs; resolution must be fresh each run (matches MCP's per-run lifecycle).

## 5. Approval gating

### D49 — MCP tools are gated by default in interactive runs
The interactive gate hardcodes `{bash, write_file, edit_file}` (`bridge.py`). For a run that
loads MCP servers, **union the run's MCP tool names into the gated set**, so every external
tool call pauses for human approval (the existing SSE `ToolApprovalRequested` flow).
Autonomous (Overleaf, D19) runs keep the gate **off**, unchanged.
- **Why:** an MCP server is arbitrary external code (a stdio server is literally an arbitrary
  local `command`); its tools can read/write/network. They are *less* trusted than the
  built-ins, so they must be **at least** as gated. Defaulting to gated is fail-safe.
- **Implementation:** the gated set becomes `BASE_GATED ∪ resolved_mcp_tool_names` for the
  run (the names from D48 step 5). No change to the gate callback shape (`gate(name, args) → bool`).
- **Rejected:** *ungated MCP tools (today's behavior if we just populated the dict).* Silently
  grants a freshly-added server full unattended action on the first run — the wrong default.
- **Open:** a per-server "trusted, don't gate" opt-in is a plausible later refinement
  (Open Q-2) — default-gated now, allow-listing later.

## 6. Schema & API surface

### D50 — Additive tables + CRUD/assignment routes; no derived state to drift
- **Schema:** `skills` + `skill_projects`, `mcp_servers` + `mcp_server_projects` (above). A
  `db.py` migration adds them; `store.py` gets the queries (list, get, upsert, delete,
  assign/unassign, and the two **resolution** reads keyed by `project_id`).
- **API (`routes/skills.py`, `routes/mcp_servers.py`):**
  - `GET /api/skills` · `POST /api/skills` · `GET/PUT/DELETE /api/skills/{id}`
  - `PUT /api/skills/{id}/assignment` `{is_global, project_ids[]}`
  - `GET /api/mcp-servers` · `POST` · `GET/PUT/DELETE /api/mcp-servers/{id}` (specs **redacted**
    on read) · `PUT /api/mcp-servers/{id}/assignment`
  - Optional read-side: `GET /api/projects/{id}/attachments` → the skills + servers that
    resolve for that project (feeds the project-window "attached" indicator, UI D54).
- **Why additive + index-only:** nothing here is derived-from-live-state (unlike graph status);
  it's user-authored catalog data, so it's stored plainly. Resolution (D48) is the only derived
  step and it's recomputed each run, so it can't go stale.
- **Guards:** MCP secret fields validated + redacted (D46); skill `slug` uniqueness; assignment
  routes reject unknown `project_id`s; deletes cascade the join rows.

## 7. Import from GitHub (the primary Add path)

### D56 — Add-from-GitHub: fetch via shallow clone, snapshot into the row, keep `source_url`
The headline Add gesture is **paste a GitHub URL → Add → choose scope**. The adapter normalizes
the URL (repo, `…/tree/<ref>/<path>`, `…/blob/<ref>/<path>/file.md`, a `raw.githubusercontent.com`
URL, or a gist) into `(clone_url, ref, subpath)`, **shallow-clones** to a tempdir
(`git clone --depth 1 --branch <ref>`), reads the relevant file(s), and writes the result into a
normal `skills` / `mcp_servers` row. Two new nullable columns on **both** tables record provenance:
`source_url` (the link) and `source_ref` (branch/commit), enabling a later **re-sync from source**.
- **Skills (one-click):** locate the skill markdown in the clone — the `subpath` file if given,
  else `SKILL.md`, else `README.md`, else the first `*.md` — and store it as `body`; derive
  `name`/`slug` from the repo/dir (user-editable). `POST /api/skills/import {url, is_global,
  project_ids[]}` does fetch + create + assign in one call.
- **Auth + reuse:** private repos and GitHub rate limits use the **`github_token`** (D34/U3) —
  injected into the clone URL only, never persisted, exactly like the 6b push helper. **Public
  repos import without a token**, so this is *not* blocked by 6b being deferred (logged in §11).
- **Why clone, not the GitHub API:** the project already shells `git` everywhere (`gitstore.py`);
  a shallow clone handles any repo layout, private auth, and gist uniformly, and reads files
  straight from the working tree — no per-URL API parsing or pagination.
- **Why snapshot + keep the URL:** the row is self-contained (a run never needs network), but
  `source_url` lets a user pull upstream updates on demand. No live/auto-sync (a skill mustn't
  change under a run without the user's say).
- **Rejected:** *reference-only (store the URL, fetch at run time).* Makes every run network- and
  uptime-dependent and unauditable; snapshotting is the D45 "row is the source of truth" rule.
- **Guards:** clone depth 1 + size cap + timeout; tempdir removed after import; the imported
  markdown is data (injected as prompt text), never executed.

### D57 — MCP import is prefill-then-confirm, never auto-run
A GitHub repo is an MCP server's **source, not a runnable spec** (there's no universal manifest).
So `POST /api/mcp-servers/import {url}` clones, **derives a draft spec**, and returns it for the
user to **confirm/edit** before save — it does **not** silently create a live server.
- **Derivation:** read a manifest if present (`mcp.json` / `server.json` / `.mcp.json`, or
  `package.json` `bin`, or a `pyproject`/`README` run line) → `{command, args}`; else infer from
  repo type (Node → `npx -y <pkg>`, Python → `uvx <pkg>` / `python -m <module>`). Whatever is
  found pre-populates the spec form (D53); unknowns are left for the user.
- **Why confirm is mandatory (not friction):** the launch command is **arbitrary local exec**
  (D49's trust-boundary note) — importing it unconfirmed would let a pasted link run code on the
  next run. The confirm step *is* the security control on the launch command; the per-call gate
  (D49) covers the tool calls after it. So MCP has **two** trust gates: confirm-on-import (the
  command) + gate-on-call (the tools).
- **Rejected:** *one-click MCP import that auto-creates a runnable server.* Unsafe (pasted-link →
  code-exec) and unreliable (no manifest standard). Prefill-then-confirm is both safer and honest
  about what a repo actually provides.

## 10A. Deferred follow-ups (known gaps)

- **The entire MCP Factory is deferred** (Group X / D46/D49/D53/D54/D57). The Skill Factory ships
  first; MCP is designed and parked. When it resumes it inherits the scope model (D47), the run-time
  resolution seam (D48), the gating approach (D49), and the `ghimport` skeleton (D56) — so it's an
  additive build, not a redesign.
- **Agent-assisted MCP setup** (considered, parked with MCP). Idea: instead of deterministic
  prefill (D57), let the agent clone the repo, read the README, install, and **validate** the server
  (start it, confirm `list_tools`) → a known-working spec. Strong fit for the agent and solves the
  no-manifest long tail. **Blocker:** it executes untrusted repo code (`npm install` postinstall,
  build scripts) **at import time** — earlier + broader than D57's confirm-on-launch — so it needs a
  **sandbox** (or at minimum the per-command approval gate) before it's safe. Trust model would be
  three-layer: sandbox setup → confirm the spec → gate the calls (D49). Revisit when MCP resumes;
  the sandbox-vs-gate decision is the gate on it.
- **Per-session skill/MCP picker for loose sessions** (follow-up to D47). The factory model
  resolves nothing for loose sessions by design; a later manual picker can opt already-authored
  units into a one-off session. Non-blocking; the catalog + project resolution come first.
- **Per-server "trusted, don't gate" opt-in** (follow-up to D49). Default-gated now; an
  allow-list to skip the gate for a vetted server is a later refinement.
- **MCP tool-name display / alias control** (follow-up to D46). The prover auto-prefixes only on
  clash; a user-facing alias is deferred.

## 11. Open questions

1. **Does MCP mirror skill scoping exactly?** Assumed **yes** (D47 is written symmetric) — same
   `is_global` + per-project join + loose-gets-none. Confirm before X1; if MCP should scope
   differently, only §3/§4 change. *(confirm at X1)*
2. **Per-server gate opt-out (D49):** default-gated is decided; whether to add a "trusted" flag
   now or defer. *(lean defer — see §10A)*
3. **Secret storage at rest (D46):** reuse the provider-key storage exactly (env-exported /
   config) vs. encrypt-at-rest in the DB. *(decide at X1 — match whatever the github_token/U3
   work lands on, for one consistent secret story)*
4. **Resolution precedence/dedupe:** if a skill is both global *and* explicitly assigned, or two
   servers register the same tool name — dedupe by `slug`/`name`, prover handles tool clashes
   via `<server>__` prefix. *(low-stakes; settle in W3/X3)*
5. **GitHub import depends on the `github_token` for private/rate-limited repos (D56), but that
   token is part of 6b/U3, currently deferred** (the `SettingsPage.tsx` fix). Public-repo import
   works tokenless, so W4/X4 can land first; private-repo import lights up once 6b ships. *(land
   public import now; gate the private path on 6b)*
6. **Re-sync from source (D56):** an on-demand "update from `source_url`" action — in scope for
   v2.1.1 or a follow-up? *(lean: ship import now, add re-sync as a fast follow)*
