## 1. Vision

v2.1 gave the app two top-level surfaces: the loose **chat** view and the **project window**
(D36 — it replaces chat+canvas when a project is open). v2.1.1 adds a **third**: the
**Factory** — a place to author reusable capabilities *once* and attach them to the projects
that want them.

```
┌──────────┬──────────────────────────────────────────────┐
│ SIDEBAR  │   FACTORY PAGE  (when a factory is open)       │
│          │  ┌────────────────────────┬─────────────────┐ │
│ Projects │  │  catalog list          │  view / edit:    │ │
│  · Epsi… │  │  (skills | servers)    │  · skill markdown │ │
│          │  │                        │  · server spec    │ │
│ Library  │  │  [+ New]               │  + ASSIGNMENT:    │ │
│  · Skills│  │                        │  global / projects│ │
│  · MCP   │  │                        │                  │ │
│ Chats    │  └────────────────────────┴─────────────────┘ │
└──────────┴──────────────────────────────────────────────┘
```

The motivating loop: a mathematician writes a "how we prove inequalities here" skill (or wires
up an MCP server for literature search) **once**, marks it global or attaches it to a couple of
projects, and every run in those projects picks it up automatically. The page is deliberately
**simple** — list on the left, view/edit + assignment on the right. (Backend resolution D48
does the rest, invisibly, at run start.)

## 1a. Foundational principles

Carry v2.1's principles (P3 tabbed project window, P4 graph-as-navigation, P5 three-docs-one-editor)
and add two.

### P6 — A factory is a *library*, not a settings pane
Skills and MCP servers are first-class authored objects with their own list, editor, and
lifecycle — not buried checkboxes in Settings. They get a sidebar **Library** group and a
full-page surface, parallel to Projects. (Secrets *within* an MCP server still reuse the
Settings redaction UX, D43, but the catalog itself is a library.)

### P7 — Assignment is the same gesture everywhere
Attaching a skill and attaching an MCP server is **one reusable control** (P5's spirit, applied
to scope): a **global toggle** + a **project multi-select**. Authored once as a component, used
on both pages. Mirrors backend D47's single scope model so the UI can't imply a distinction the
backend doesn't have.

## 2. Information architecture

Extends `v2.1-ui-design.md` §2:

- **The sidebar gains a `Library` group** (between Projects and Chats): two entries, **Skills**
  and **MCP**, each opening its factory page (D51). Like a project, a factory page **replaces**
  chat+canvas; the sidebar stays. (Routing reuses D36's view switch in `App.tsx`.)
- **A factory page is a two-pane catalog**: list (left) + detail/editor + assignment (right) —
  the same skeleton for both factories, specialized by unit type (D52/D53).
- **The project window shows what's attached** (read-only): an "Attached skills / servers"
  indicator in the project window so a project's effective capabilities are visible where the
  work happens (D54), without leaving the project to check the factory.
- **Loose chats show nothing factory-related** — matching backend D47 (loose = none). No
  attachment UI on a loose session in v2.1.1 (the per-session picker is deferred, backend §10A).

## 3. The factory pages + decisions

### D51 — Skills & MCP are sidebar `Library` surfaces (parallel to Projects), not Settings tabs
A new **Library** sidebar group lists **Skills** and **MCP**; clicking either opens its factory
page (full surface, replacing chat+canvas via the D36 routing switch). `App.tsx` view state gains
`factory:skills` / `factory:mcp`.
- **Why:** these are authored libraries with real CRUD + assignment, not toggles. Giving them
  sidebar peers to Projects matches their weight and keeps Settings about *config*, not *content*.
- **Rejected:** *tabs inside Settings.* Cramped, and conflates "API keys / preferences" with
  "author a reusable skill." *A tab inside each project window* — wrong scope; a factory unit is
  cross-project, so it can't live inside one project.

### D52 — Skill Factory page: catalog list + `MarkdownDoc` editor + assignment
Left: the skills list (name + a global/▣-projects badge). Right: the selected skill — **name**,
the **markdown body** edited with the **existing `MarkdownDoc` component** (D39, incl. its
expand-to-modal for long bodies), and the **assignment control** (D55). `[+ New skill]` creates
an empty one.
- **Why reuse `MarkdownDoc`:** a skill *is* a markdown doc; the component already does
  view→edit→save + the modal. No new editor.
- **Rejected:** *a structured form.* Skills are free prose; markdown-first matches how the `.lea`
  docs are authored (P5).

### D58 — "Add from GitHub" is the headline Add control (paste link → scope → Add)
Each factory page leads with a compact **Add from GitHub** affordance: a **URL input** + the
shared scope control (Global / select projects) + an **Add** button. For **skills it's one step** —
paste → choose scope → Add fetches the markdown and the new skill appears, selected, ready to
tweak (`POST /api/skills/import`, D56). A small **"or write one from scratch"** link opens an empty
editor for the manual path. (The MCP variant — prefill-the-spec-form then confirm, D57 — ships
with the deferred MCP page.)
- **Why lead with it:** skills are shared as repos; pasting a link is how a user actually gets one.
  Making import the primary gesture (not buried behind "New → advanced") is the "very easy" ask.
- **States:** fetching (spinner on Add), success (skill selected), error (bad URL / not found /
  private-without-token → inline message, with a hint that a token in Settings unlocks private
  repos — 6b). Empty catalog shows the Add control front-and-center.
- **Rejected:** *a multi-step import wizard / modal.* Overkill for "paste a link"; the inline
  input keeps it one gesture. *Auto-importing on paste* — Add stays an explicit click.

### D53 — MCP Factory page: catalog list + spec form + redacted secrets + assignment  *(deferred)*
Left: the servers list (name + transport badge + global/▣-projects badge). Right: a **spec form**
that switches on transport — **stdio** (`command`, `args[]`, `env` key/value, `cwd`) or **remote**
(`url`, `headers` key/value, `transport`: sse | streamable_http) — plus the **assignment control**.
Secret-bearing fields (`env`/`headers` values) use the **redacted field UX from Settings** (D43):
presence-only, never echoed, re-enter to change.
- **Why a form (not raw JSON):** the prover spec is structured; a form prevents malformed specs
  and gives secrets the redaction treatment a JSON blob can't. (Backend D46 stores `spec_json`;
  the form is just its editor.)
- **Why reuse the D43 redaction UX:** MCP `env`/`headers` carry tokens; they must never round-trip
  to the client — exactly the provider-key/`github_token` contract.
- **Rejected:** *free-text JSON editor.* No validation, no redaction, easy to footgun.

### D54 — The project window surfaces what's attached (read-only)  *(deferred with MCP)*
The project window gets an **"Attached" indicator** (a small section/rail entry) listing the
skills + servers that resolve for that project (`GET /api/projects/{id}/attachments`,
backend D50). Read-only here — you *manage* assignment in the factory; you *see* the effect in
the project.
- **Why:** a run's effective capabilities should be visible where the work is, not only by
  cross-referencing the factory. Closes the "what's actually active here?" loop.
- **Rejected:** *full assignment editing inside the project window.* Two places to edit the same
  relation invites drift; keep authoring in the factory, viewing in the project.

### D55 — One `ScopeAssignment` control (global toggle + project multi-select), used on both pages
A single reusable component: a **Global** toggle (= all projects) and, when off, a **project
multi-select** (the projects this unit attaches to). Emits the backend assignment payload
(`{is_global, project_ids[]}`, D50) on save.
- **Why:** P7 — assignment is the same gesture for skills and servers; one component guarantees
  the UI mirrors the single backend scope model (D47) and can't imply a phantom difference.
- **Rejected:** *separate per-page assignment widgets.* Duplicate code, risk of divergent
  semantics. *A per-project "add skill" flow* — inverts the natural authoring direction (you
  author the unit, then choose where it applies).

## 8. State & wiring

- **`stores/factories.ts`** — skills + servers lists, selection, CRUD + assignment actions
  (mirrors `stores/projects.ts` D35; no prop-drilling). Loads on first factory open.

### 8.3 Touch points (existing files)

  + attachments fetchers

### 8.4 Reuse (no change)
- **`MarkdownDoc`** (D39) for skill bodies — incl. the expand-to-modal editor.
- **The Settings redacted-field pattern** (D43) for MCP `env`/`headers` secrets.
- **The project-window routing** (D36) — a factory page is just another full-surface view.

## 9. Open design questions

1. **MCP spec form vs. advanced raw-JSON escape hatch** — the form covers stdio/remote; a
   power-user "edit raw spec" toggle could come later for exotic specs. *(defer — form-first)*
2. **Per-session skill picker (loose sessions)** — backend defers it (§10A); the UI hook would be
   a small "Skills" affordance on a loose session's composer. *(defer with the backend)*
3. **Attachment indicator placement** — a rail card vs. a line in the Overview hero vs. a small
   tab. *(decide while building F13)*
4. **Empty/first-run states** — the factory pages need friendly empty states ("No skills yet —
   author one to share house-rules across projects"). *(settle during F11/F12)*
