# LeaUI v2.3 — Concurrent runs (remove the single-session restriction)

**Status:** design · **Decisions:** D69–D84 · **Predecessor:** `v2.2-live-editor-architecture.md` (D59–D68)

> D79–D84 (§4) refine the subagent/prompt decisions D76–D78 against two shipping
> reference implementations — **opencode** (`TaskTool`, permission rulesets, markdown
> agents) and **OpenHands SDK** (`DeclaredResources`/`ResourceLockManager`, skills with
> keyword/path triggers, section-composed prompts). They are the concrete mechanics
> under the D76–D78 prose. Prompt sources to copy are catalogued in the appendix (§7).

---

## 1. The problem

Today one person can formalize exactly one thing at a time. Open four chats and the
second one dies:

```python
# adapter/app/routes/runs.py:180-185
active_run_id = bridge.current_active_run_id()
if active_run_id is not None and active_run_id != run_id:
    raise HTTPException(status_code=409, detail="Another Lea run is already active.")
```

That 409 is not a setting to flip. It is the visible edge of a global mutex:

```python
# adapter/app/bridge.py:58-62
# Only one Lea activation may run at a time — they share the on-disk workspace
# and the warm LSP daemon. ...
active_run_lock = Lock()
```

The comment is the entire safety argument for the system, and **both halves of it are
load-bearing**. Removing the lock without further work does not give you concurrency.
It gives you silently wrong proof verdicts.

This matters twice over: for its own sake (multiple chats), and because it is the
prerequisite for the multi-agent work — subagents mean many prover activations in
flight, which trips exactly the same shared state.

## 2. What the lock is hiding

### 2.1 Wrong-answer hazards (the serious tier)

**H1 — The Lean daemon returns another session's verdict.**
Every session's proofs live under one Lake root (`prover/workspace/lakefile.lean` is
the only lakefile), so all sessions share **one** `LeanDaemon` and one `lean --server`.

`check_via_lsp` (`prover/lea/lsp_daemon.py:267-283`) holds `_lock` only around the
*registry lookup*. The actual check — `d.check(file_path, content)` on line 283 —
runs **outside the lock**. Inside `check()`:

- on entry it **destructively purges** the single shared `self.queue` (`:109-114`),
  discarding diagnostics another thread is still waiting for;
- its wait loop **drops** any message whose URI doesn't match instead of requeuing it
  (`:147-172`) — `Queue.get` is destructive and nothing puts it back.

Failure modes:
- *Different files* (the common case): the two threads destroy each other's frames,
  neither gets a definitive answer, both spin to `_CHECK_TIMEOUT` (**900s**), then
  `tools.lean_check`'s bare `except Exception: pass` (`tools.py:177`) silently falls
  back to a cold ~88s subprocess. A check that should take 0.2s takes 15 minutes.
- *Same file*: the URI filter matches, and thread A **returns a verdict computed from
  thread B's file content**. That feeds `_lean_check_has_error` → `ProofVerificationState`
  → the final gate → a false **"Proved."**

Also in the same file: `self.opened` TOCTOU (`:117,:133`), `self.calls` non-atomic
read-modify-write used as the LSP document version (`:106-108`), an unsynchronized
`_send` on an **unbuffered** pipe (`:212-214`, `bufsize=0`) that can interleave
`Content-Length` headers with bodies and *permanently desync the framing*, and a
stale-restart path (`:272-282`) that calls `d.shutdown()` while another thread is
mid-`check()`.

**H2 — SafeVerify cross-contaminates.**
`prover/lea/interface.py:131-152` keys its scratch files by the **file stem** only, in
one shared `.sv_scratch` dir, and deletes them in a `finally`:

```python
stem = p.stem or "proof"
target     = scratch / f"{stem}_sv_target.lean"
submission = scratch / f"{stem}_sv_submission.lean"
```

Two sessions both working on `Div6.lean` verify **A's submission against B's target** —
a wrong `ok`/`rejected` on the strongest claim the system makes. The `finally` unlink
also deletes files the other run is mid-compile on. Same defect one layer down in
`safeverify.py:204-239`. Note `interface.verify` never passes `scratch_dir=` even
though the parameter exists.

### 2.2 Wrong-behavior hazards (shared project repos)

Loose sessions each own `proofs/<session-id>` — safe. But **project** sessions all
resolve to the same repo (`projects.py:142-149`), and that is the normal Overleaf case:
every theorem in one document is a session in one project.

**H3 — `git add -A` cross-stages.** `gitstore.py` has **no locking at all**.
`commit_all` (`:96`) / `commit_write` (`:107`) do a repo-wide `git add -A` then a
three-process `diff --cached → commit → rev-parse HEAD` sequence. Run A stages run B's
half-written file; A's `code_step.commit_sha` points at a tree containing B's content;
`rev-parse HEAD` can return B's SHA; and `.git/index.lock` collisions throw
`GitStoreError` into run threads.

**H4 — The agent is told a peer's theorem is "the human's edit".**
`bridge.py:238` `_divergence_context` diffs the **whole repo**
(`gs.diff(repo_key, agent_step["commit_sha"], "HEAD")`). Once a peer session commits
into the shared project repo, the next run in *this* session is handed that diff
prefixed with *"The human edited the proof files since your last turn."* The agent may
then go "acknowledge" and edit someone else's file. Same defect on the human
canvas-save path (`gitstore.py:186` `uncommitted_diff` + `routes/sessions.py:149`) —
and the D62 modal lock is *per-session*, so it does not stop a human saving in session
A while an agent drives session B of the same project.

### 2.3 Infrastructure hazards

**H5 — SQLite.** `db.py:18-27` opens a fresh connection per call with
`check_same_thread=False`, but there are **no PRAGMAs anywhere** — no WAL, no
`busy_timeout`. Journal mode is DELETE and transactions are DEFERRED. `store._next_seq`
(`store.py:994`) does a `SELECT max(seq)+1` and then an `INSERT` on the same
connection: that is a **lock upgrade**, and SQLite returns `SQLITE_BUSY` *immediately
without invoking the busy handler* (it would deadlock otherwise). So
`database is locked` gets thrown into run threads. Adding `busy_timeout` alone **does
not fix this.**

**H6 — Lock leak (pre-existing).** The lock is acquired at `bridge.py:317` but the
`try:` block only starts at `:407`. Anything in between (`gs.init_repo`,
`repo_for_session`, `materialize_project_skills`, lines 335-371) that throws leaks
`active_run_lock` **permanently** — every later run 409s until the process restarts.

**H7 — Interrupt heuristic.** `runs.py:158`: a `pending` run whose id isn't the scalar
`_active_run_id` is force-failed. With N runs that scalar can name at most one, so it
would wrongly fail other genuinely-live runs.

**H8 — Cold-fallback herd.** `lean_check` swallows *any* LSP exception and falls back
to a cold `lake env lean` (~88s, full Mathlib). The run lock currently caps that at one.
Under N runs, a single LSP hiccup spawns N Mathlib-loading processes.

**H9 — SafeVerify was relying on the lock for throughput, not just correctness.**
`interface.verify`'s own docstring: *"Expensive (~two compiles + a replay), so the
adapter runs it once on the final artifact and **serializes calls**."* That
serialization **is** the run lock. It is called from a sync route
(`routes/sessions.py:250`), so FastAPI runs it in the threadpool — N concurrent
verifies become 2N Mathlib compiles the moment the lock goes.

**H10 — `lake build` on the shared workspace.** `tools.rebuild_module` runs
`lake build` with `cwd=lake_root`. `docs/PLAN-edit-save-latency.md:231` documents this
as a deliberate invariant: *"the Lake workspace is shared, and concurrent `lake build`
invocations against it are exactly what the serial loop exists to avoid"* — and lists
parallel `lake build` as an explicit **non-goal**.

**H11 — Shared cwd.** `tools.py:338` `bash` runs `subprocess.run(shell=True)` with **no
`cwd=`**, inheriting the process-global cwd. Two runs in one project also literally
share `working_dir` (`bridge.py:436`).

**H12 — MCP tool registry (dormant).** `registry.py:42-43` `REGISTRY`/`_ORDER` are
process-global and mutated per activation by `mcp.py:146,150-153`. Two concurrent
MCP-enabled activations corrupt each other's toolsets (A's `stop()` unregisters B's
tools by name). The adapter never sets `mcp_servers` today, so this is dormant — but it
becomes live the moment it isn't.

### 2.4 Things that are already fine

Worth stating so nobody "fixes" them: `_stop_events`, `_pending_approvals`, and
`_session_allowlists` are plain dicts keyed by run_id/session_id with single-op
mutations — GIL-atomic and correct under N runs. `agent.py` has no mutable globals, no
`chdir`, no environ writes, no disk session state (D16: one activation = one pure call).
The Mathlib search budget is **prompt text only** (`prompt.py:158`), not a shared
counter. `providers.py` caches no clients.

## 3. The key insight — Lean already solved this

The Lean 4 server is **not** a synchronous request/response loop. It is
[a watchdog process coordinating one file-worker process **per open file**](https://github.com/leanprover/lean4/blob/master/src/Lean/Server/README.md),
and the design explicitly states that *"file processing and requests+notifications
against a file should be concurrent"* — partly so requests stay cancellable, partly for
fault isolation (*"corruption of a single per-file worker cannot affect the stability of
the whole server"*).

Every real LSP client — vscode-lean4, lean.nvim, coc-lean — does what JSON-RPC requires:
a reader loop dispatching responses by request `id` and notifications by `uri`. It never
assumes one request is in flight. That is why nobody else hits H1.

**Lea's `lsp_daemon.py` is not an LSP client.** It is a synchronous shim written against
a mistaken model of the protocol. So making it multiplex is not a risky new design — it
is *correcting* the component, in the direction every other Lean client already goes.

Consequence: one server, **one resident Mathlib**, N open documents, N parallel file
workers. Sessions own distinct canonical files, and subagents check distinct temporary
candidate files. That gives Lean distinct URIs for the concurrent work; only the
coordinator writes the clean canonical artifact.

## 4. Decisions

**D69 · Concurrency is keyed per session; one run per session.**
A second run for the same session supersedes the incumbent (today's behavior, now
scoped to the session instead of the whole server). Rationale beyond the obvious:
`store.latest_transcript_for_session` (the D16 multi-turn replay) and
`_divergence_context` both assume a single writer per session. Subagents later run
*inside* one run — the registry is already shaped for that (`ActiveRun.activations`) —
and the parent run remains the single committer to the session's canonical file(s).

**D70 · Keep the 409; redefine it as "at capacity."**
The Overleaf companion **already** fires runs concurrently (`server.mjs:333`,
fire-and-forget, deduped only per theorem `jobKey`) and **uses the 409 as its queue**
(`leaApiClient.mjs:564-633` — retry with jitter, consult the run row, adopt a terminal
outcome). Its comment states the assumption verbatim. Redefining 409 as "at capacity"
is a semantic no-op for it: **zero companion changes**, and the Overleaf suite staying
green *without modification* is the proof we got this right.

Rejected: a server-side queue that accepts the SSE attach and emits a `queued` event.
It pins a connection + task per waiter, makes run ownership ambiguous if the client
disconnects mid-wait, and eats the browser's 6-connections-per-origin budget (the UI
already permanently holds `/api/sessions/events`). The 409 keeps the invariant crisp:
**exactly one attach ever spawns a runner; the run row stays `pending` and is startable
by anyone.**

Corollary: removing the 409 *without* server-side admission control would be strictly
worse than the status quo — a 20-theorem Overleaf document would fire 20 simultaneous
prover runs.

**D71 · `lsp_daemon.py` becomes a real LSP client.**
Reader thread dispatching by `id` (responses) and `uri` (notifications) into
**per-request queues**. The shared `self.queue` is deleted, and with it the destructive
purge and the message-dropping. Plus: `_io_lock` around `_send`; `_state_lock` for
`opened` / version / `calls`; a **per-URI lock** for the same-document case; and
**leases** so a stale daemon is removed from the registry (no new leases reach it) and
shut down only when its last in-flight check drains.

Same-URI is solved by a lock, **not** by versions: a Lean file worker holds one document
state per URI, so two contents for one URI cannot be checked concurrently by
construction. The lock costs nothing in practice — different sessions use different
files.

Rejected: adopting `leanclient`. The observability fork does
(`../lea-prover-observability/lea/lsp_manager.py`) — and wraps every check in a coarse
`threading.Lock` (`:20,:43`), re-introducing exactly the serialization we're removing.
It also does open → diagnostics → **close** per call, discarding the warm-document model
that makes rechecks 0.2s. And it's a new dependency in a vendored `uv` package with a
pinned toolchain. Fix in place; it's a 317-line file and the transport rewrite is ~150.

**D72 · Concurrent runs are allowed inside one project repo, but paths are owned.**
The intended project shape is one theorem/definition cluster per session, usually one
canonical `.lean` file. Serializing per project would deliver zero concurrency for the
primary Overleaf use case (all N theorems of one document live in one project). Instead:
each active session gets an exclusive canonical Lean path, and Git operations are scoped
to that path.

`GitStore` keeps a **per-repo lock** around the whole multi-process commit sequence, but
commits use **path-scoped `git add -- <paths>`** so a run's commit contains only its own
canonical output. Divergence checks are path-scoped too: a peer session's theorem is not
"the human edited your proof"; it is just a sibling file changing in the same project.

Rejected: a git **worktree** per run — it would have to live under `workspace/proofs/`
(that's `lean_lib Lea where srcDir := "proofs"`) or Lean can't resolve
`import Lea.<Project>.<Module>`, and a worktree at `proofs/Lea/<Project>__run7/` changes
every module name. Rejected: storing proof snapshots only in SQLite — Git already gives
durable history, diffs, exports, and a project filesystem; the problem is repo-wide
staging/diffing, not Git itself.

Path ownership is enforced at the adapter/tool boundary. A session may write its owned
canonical file(s) plus its run's temporary candidate directory. Writes to another live
session's canonical path are denied. This turns "one lemma -> one file" from a prompt
convention into a concurrency invariant.

**D73 · SQLite: WAL + explicit `BEGIN IMMEDIATE` for writers.**
`isolation_level=None` (explicit transaction control) and `BEGIN IMMEDIATE` on every
mutating connection, so `_next_seq`'s read-then-write holds the write lock across both.
`busy_timeout` is added but is **not** the fix (see H5). WAL additionally means readers
never block writers, so the `/api/sessions/events` digest poll and the companion's retry
reads stop contending with run threads.

This is not a throughput optimization. SQLite writes are fast, but concurrent runs make
the read-then-insert timeline sequence real concurrency. The explicit writer transaction
makes the sequencing deterministic instead of "rarely flaky."

`UNIQUE(session_id, seq)` on `code_steps` alone would be *wrong* — `messages` and
`code_steps` share one counter (C4), a constraint spanning a UNION that SQLite can't
express. `BEGIN IMMEDIATE` alone makes the existing `max()+1` correct. Optional
follow-up: a `session_seq` counter table (`db.py:216` says a schema change just means a
fresh DB, so it's free).

**D74 · Bound the expensive fallbacks.** Semaphores on the cold `lean_check` subprocess
path (H8) and on `safeverify.verify_proof` (H9); a global `lake build` lock keyed by
lake_root with `mark_stale()` called inside it (H10). `lean_check` via the LSP daemon is
a *different process*, so it stays parallel with builds — that's the win.

**D75 · The UI keeps ONE live stream.** No N-EventSource rework. Everything a background
run emits is already persisted (git for bytes, SQLite for metadata) and re-derivable from
`GET /api/sessions/{id}` — that is the entire point of D7/D8/D14. The open session keeps
its single live stream; background runs are tracked through the **`sessions_changed`
stream that already exists**. `store.py:164-169` *already computes* `active_run_count`
and then `pop()`s it at `:186`; `sessions_digest` (`:200-214`) already includes
`active_runs`. So the sidebar fix is ~3 lines and needs no new server stream.

The only thing given up: token-by-token streaming for a session nobody is looking at.

**D76 · Subagents are child activations with scoped tools.**
Within a run, subagents do not mutate the session's canonical Lean file directly. They
write and check temporary candidate files under an ignored per-run tree, e.g.
`.lea/tmp/<run_id>/<agent_id>/Scratch.lean`, then return checked candidates, imports,
helper lemmas, failures, and diagnostics to the parent/coordinator. The parent collates
the useful results and commits the clean artifact to the session's canonical path.

This makes same-lemma subagent concurrency tractable without CRDTs or character-level
merge theory. The expensive work stays parallel (search, proof attempts, Lean checks on
distinct URIs); the canonical file has one writer. If multiple candidate patches target
the same output, the parent uses optimistic concurrency control: accept against the
current base, re-check, and rerun/rebase/drop stale candidates.

Implementation support: `prover/lea/runctx.py` carries `working_dir`, `run_key`, and
`candidate_dir` as `ContextVar`s set in `agent._run_events_inner`. This fixes `bash`
cwd (H11), gives SafeVerify per-activation scratch keys, and is the later path to a
ContextVar-scoped tool registry snapshot instead of the module global (H12).

The shape is borrowed from OpenCode's `TaskTool`: subagents are ordinary child sessions
or activations, selected by name, with their own prompt/model/step budget and a derived
permission ruleset. The important part is not the UI; it is that mutating tools enforce
permissions at execution time. Lea's existing `write_file` / `edit_file` approval seam
becomes the hard ACL:

| Agent | Mode | Permissions | Purpose |
|---|---|---|---|
| `formalizer` | primary | canonical write, commit, SafeVerify, spawn approved subagents | Parent/coordinator for a theorem session |
| `plan` | primary | read-only, no canonical writes | Decompose theorem, choose strategy, review design before execution |
| `premise-search` | subagent | read/search only | Find Mathlib/project lemmas, imports, and proof patterns |
| `proof-candidate` | subagent | write only `.lea/tmp/<run>/<agent>/**`, `lean_check` | Try proof candidates in isolated Lean files |
| `critic` | subagent | read-only | Explain diagnostics and suggest next attempts |
| `refactor-candidate` | subagent | write only temp paths, `lean_check` | Explore helper lemmas/definitions or file splits |

Default depth is 1: subagents cannot spawn subagents unless explicitly enabled. Each
subagent gets a max step/check/token budget. Candidate outputs are trusted only as
evidence: the parent must re-check the promoted canonical file and final SafeVerify is
unchanged.

Two hidden summarizers have different jobs:

- `run_summary`: short human-facing recap for the UI/project timeline, like a PR
  description.
- `proof_context_compaction`: machine-facing checkpoint for future turns: theorem
  statement, current canonical file state, tried lemmas, failed proof paths, useful
  imports, candidate results, and remaining obligations.

**D77 · Parallel tools declare resources and conflict at the tool boundary.**
OpenHands' useful concurrency trick is not a global scheduler; it is
`declared_resources()`. A batch of tool calls can run in parallel, but the executor
serializes only calls that touch the same resource. Its file editor declares a
normalized `file:<path>` lock even for reads, and its terminal declares
`terminal:session` unless the terminal backend provides pane isolation.

Lea should use the same rule, specialized for Lean:

```text
lean_file:<canonical-path>       # canonical write/read conflicts
candidate_file:<run>/<agent>/<path>
lsp_uri:<uri>                    # same Lean document cannot check two contents
lsp_workspace:<lake-root>        # daemon lifecycle / restart operations
git_repo:<repo-key>              # commit sequence only, not all proof work
lake_build:<lake-root>           # rebuilds stay serialized
safeverify:<lake-root>           # bounded expensive final verification
```

This gives the database-theory property we want: non-overlapping candidate files,
searches, and Lean checks proceed concurrently; same-file writes serialize; repo-wide
operations are explicit and rare. Tools that do not declare resources default to a
tool-wide lock until they are audited. Resource locks do not replace path ownership
(D72/D76); they are the executor-level enforcement mechanism that makes ownership
composable when subagents arrive.

**D78 · Subagent profiles are declarative; transcripts are debug artifacts.**
OpenHands' Markdown agent definitions are a good shape for Lea even though its runtime
is too generic to adopt wholesale. Lea should keep a small declarative profile format
for primary agents and subagents, for example:

```yaml
name: proof-candidate
mode: subagent
model: inherit
tools: [read_file, write_candidate, lean_check]
write_scope: ".lea/tmp/{run_id}/{agent_id}/**"
permission_mode: never_confirm
max_steps: 12
max_checks: 8
max_tokens: 12000
condenser: proof_context_compaction
```

Profiles are loaded by the adapter/prover at activation start and converted into the
existing tool registry/approval policy. They are not a second plugin system. Built-ins
cover `formalizer`, `plan`, `premise-search`, `proof-candidate`, `critic`, and
`refactor-candidate`; project overrides can come later only after the hard ACL is in
place.

Child activation transcripts are stored for debugging/resume, but they are not normal
proof history. A transcript can be event-log shaped (`messages`, tool calls,
observations, summaries, final result), while `code_steps` remains reserved for
promoted canonical proof evolution. The parent/coordinator promotes exactly one clean
canonical artifact into `code_steps`, with the winning candidate's transcript linked by
id for auditability.

## 4b. Refinements from the reference harnesses (D79–D84)

D76–D78 fixed the *shape*. These fix the *mechanics* — each one is the part of a
shipping implementation that makes the shape safe, lifted and specialized for Lean.

**D79 · Subagent permissions compose by tightening only.**
D76's ACL table is static; the rule that makes it *composable under nesting* comes from
opencode's `deriveSubagentSessionPermission` (`agent/subagent-permissions.ts`): a child's
effective permission is **its own capabilities ∪ the parent's *deny* rules ∪ the parent's
path/external-dir rules**. A restriction can never loosen through delegation — only tighten.
That single invariant is what keeps the `.lea/tmp/<run>/<agent>/**` write-scoping (D76)
intact when a subagent itself spawns work: the child cannot reacquire a canonical-write
capability the parent lacked. Two independent recursion guards, both adopted: a depth walk
over the parent chain (default depth 1, as D76 states) **and** `spawn_subagent:*` denied to
subagents by default, so a profile must *explicitly* opt into nesting. Enforcement point is
unchanged — Lea's `write_file`/`edit_file` approval seam is the ACL (D76); this decides what
ruleset that seam evaluates for a child.

**D80 · Resources are three-state; "undeclared" is the safety net, not an omission.**
D77 named the resource-key namespace; OpenHands' `DeclaredResources` (`sdk/tool/tool.py`)
gives the semantics that let it roll out incrementally without auditing every tool first:

- `declared=False` (the base-class default) → executor takes a tool-wide mutex
  `tool:<name>`. An un-audited tool **serializes itself automatically** — safe by default.
- `declared=True, keys=()` → "analyzed, touches nothing shared" → no lock (read-only
  `#check`, goal/term inspection, `search_mathlib`).
- `declared=True, keys=("lean_file:<path>", …)` → lock exactly those keys.

`ResourceLockManager` acquires keys in **sorted order** (deadlock-free), uses **FIFO locks**
(no writer starvation), and applies **per-prefix timeouts** — for Lea: `lean_file` 30s,
`lake_build`/`safeverify` 300s, `lsp_uri`/`lsp_workspace` 300s — raising rather than hanging.
Each executor owns its own pool + manager so a subagent's locks can't deadlock the parent.
Mapping onto D77's namespace: the LSP-daemon `lean_check` declares `lsp_uri:<uri>` (and
stays a *different process* from `lake_build`, so builds and daemon-checks run in parallel —
that is the win D74 called out); a canonical write declares `lean_file:<path>`; SafeVerify
declares `safeverify:<lake-root>`. Anything not yet reasoned about stays `declared=False`
and is simply serialized until audited — this is the precise meaning of D77's "default to a
tool-wide lock."

**D81 · Tactic cascades are domain-scoped via path triggers.**
`_TACTIC_CASCADE` (`prompt.py:98-122`) is today one flat table for all of mathematics, so a
`Measurable f` goal and an `a ≤ b` over ℝ goal compete for the same prioritization. Borrow
OpenHands' `PathTrigger` (`sdk/skills/trigger.py`, injected at *tool-use* time by
`agent_context.get_tool_use_suffix`): the base cascade stays general, but touching a file
whose path matches a domain glob injects a domain-specific cascade — `Mathlib/MeasureTheory/`
→ measure-theory tactics (`measurability`, `fun_prop`, the relevant measure lemmas),
`Mathlib/Algebra/` → the algebra ladder, etc. Injection is at tool-use time, not in the base
system prompt, so the cacheable prefix (D82) is untouched. Mechanism already half-exists:
`skills.py` injects markdown fragments; this adds trigger metadata (keyword/path/task) to a
skill's frontmatter and a match step. Domain scoping can key off the session's declared
namespace when there is no file path yet.

**D82 · The prompt is cache-tiered: static Lean core first, volatile tail last.**
Once every role carries a per-role head (D84), prompts get longer and prompt-cache hit-rate
starts to matter (we already track `MODEL_PRICING`). Order sections the way OpenHands'
`presets.py` does — static blocks first, the single volatile block dead last — so the
cacheable prefix never shifts between turns: **shared Lean core** (`_HARD_RULES`,
`_TACTIC_CASCADE`, `_PHRASEBOOK`, `_SEARCH_BUDGET`) → **per-role head** → **volatile tail**
(workspace path, active namespace, `_divergence_context`, datetime). Today `prompt.py`
interpolates the workspace/namespace *into the middle* of the prompt; tiering moves that
retarget to the tail. Near-free, and it compounds with D81 (triggered cascades ride the
per-turn suffix, not the cached prefix).

**D83 · Subagent results are a typed envelope; the result id is the audit link.**
Both harnesses wrap a subagent's return as a structured block
(`<task id=… state=…><task_result>…</task_result></task>`) rather than dumping its
transcript, and the `bash-runner` subagent's prompt makes the contract explicit: *"the caller
does not see raw output — only what you write back."* Lea's envelope carries exactly the D76
fields — `state`, checked candidate(s), imports, helper lemmas, failures + diagnostics, short
summary — surfaced through the existing `events.py` typed stream, **not** as raw child
transcript. The envelope's `id` does triple duty: the resume handle (opencode's `task_id`,
for iterative proof attempts against the same child), and the link from a promoted
`code_step` back to the winning transcript (D78). The parent trusts the envelope as *evidence
only*: it re-checks the promoted canonical file and final SafeVerify is unchanged (D76).

**D84 · Subagent prompts *compose* the Lean core; they never replace it. (safety-critical)**
opencode's rule is that an agent's own `prompt` **replaces** the base provider prompt entirely
(`session/llm/request.ts`). Lea must **not** adopt this. Every subagent — including a
throwaway `proof-candidate` — must still carry `_HARD_RULES` (**never modify the theorem
statement**; no `sorry`/`axiom`/`native_decide`) and the tactic cascade. A role prompt that
*replaces* the base can silently drop the immutability rule and "prove" a weakened statement —
the single worst failure mode, the exact cheat SafeVerify exists to catch. Follow OpenHands'
**compositional** model instead: `shared Lean core + per-role head`, where the core is
non-overridable and the head only *adds* role framing and *narrows* tool access. Rejected:
opencode-style whole-prompt replacement — convenient, but it makes the system's core safety
invariant a per-profile opt-in, which is backwards.

Two structural notes on Phase 3:

*Admission moves into the endpoint.* Today the claim happens **inside** the spawned
thread (`bridge.py:317`) while the endpoint separately peeks at the scalar

the thread exists, also makes `interrupt`'s pending-run check **exact** rather than a

status — it just fires `onerror`. So the moment the server can 409, that is an
unthrottled request storm.

**Acceptance test** — `adapter/tests/test_concurrent_runs_e2e.py`. Six concurrent runs
across six sessions: three provable, three false. Assert **exactly** the three provable

session — this is the end-to-end version of "catch the wrong-verdict bug". Also: six
distinct loose-session repos, six distinct commit SHAs, no `database is locked`, no

**Project-concurrency guard** — `adapter/tests/test_project_path_ownership.py`. Six

diffs ignore sibling files, and attempts to write another live session's canonical path
are denied while writes under `.lea/tmp/<run_id>/...` are allowed and not surfaced as
user-visible `code_steps`.
